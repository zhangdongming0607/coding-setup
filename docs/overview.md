## 二、教学策略：集中教 vs. 分开教

### 适合集中一起教的内容

| 内容 | 理由 |
|---|---|
| 全链路心智模型（请求从浏览器到服务器的旅程） | 三人都缺，是后续所有学习的地基 |
| React 组件化思维 | 全员 5 分，庞滚滚有基础但也没系统学过 |
| Git 基础操作 | 全员 5 分，必须统一教，不然后面协作对不齐 |
| 后端入门（接口 + 数据库） | 全员 5 分，全员盲区，起点一致 |
| 代码阅读方法 | "能用 AI"到"用好 AI"的关键跳跃，集中教效率最高 |

### 适合分开教（单独辅导）的内容

| 同学 | 单独需要的 | 原因 |
|---|---|---|
| **熊** | 帮她做"减法"和优先级排序 | 什么都打 5 分，1-2 周不可能全学会，需要规划最短路径 |
| **小苗** | CSS 动画 / 交互特效方向 | 兴趣极度集中，偏视觉/动效路线，跟另外两人差异大 |
| **小苗** | 团队协作流程（Git 分支、PR、Code Review） | 目标是"参与团队开发"，另外两人暂时不需要 |
| **庞滚滚** | 后端进阶 + 独立项目指导 | 前端基础最好，节奏最稳（2-3 个月），适合独立项目持续跟进 |

## 三、教学规划

### 定位

学员已经在 vibe coding，不需要教"怎么用 AI 写代码"。要补的是底层认知：
1. AI 生成的代码**为什么**能跑
2. 跑不了的时候**怎么**定位问题
3. 项目变复杂了**怎么**组织和管理代码

参考教材：[Datawhale vibe-vibe](https://github.com/datawhalechina/vibe-vibe) 进阶篇（选读，不全读）

### 课程形式

- 每次 1 小时，每次一个核心主题
- 教练制：不是单向讲课，课上答疑 + 真实项目练习
- 每节课穿插一个 LLM 知识点，紧扣当节主题

每次课的节奏：

```
0-5min    上次作业回顾（屏幕共享/截图，暴露卡点）
5-25min   核心概念讲解（一个点，live coding 演示）
25-45min  动手环节（他们操作，屏幕共享，你巡视）
45-55min  发散问题统一回答（课上记录，课末挑 1-2 个回答）
55-60min  布置作业 + 预告下次内容
```

课堂管理：
- **发散问题**：先记到共享文档，课末统一过，挑 1-2 个当场回答，其余课后文字回复
- **暴露卡点**：动手环节要求屏幕共享或截图发群，没发的就是卡住了，主动去问

### 6 次集中课大纲

---

#### 第 1 次：全局地图 — 代码到底在哪里运行

> 解决：AI 写了一堆文件，不知道哪个在浏览器跑、哪个在服务器跑

**主线内容**
- 用学员自己做过的 vibe coding 项目拆解完整链路
- 浏览器输入 URL → DNS → 服务器返回 HTML → 浏览器渲染 → JS 执行 → 发 API 请求 → 后端处理 → 返回数据 → 页面更新
- 重点演示：Chrome DevTools — Elements 看结构、Network 看请求、Console 看报错
- 讲清楚文件分工：哪些是前端文件（组件、样式、路由）、哪些是后端文件（接口、数据库）、哪些是配置

**LLM 知识点：AI 是怎么生成代码的**
- LLM 的本质：预测下一个 token，不是"理解"你的需求
- 为什么同一个问题问两次答案不同
- 启示：AI 写的代码不一定对，你需要能验证它

**动手**：打开自己的 vibe coding 项目，用 DevTools 找出页面发了几个请求、每个请求返回了什么数据，截图标注"前端文件"和"后端接口"

**作业**：写一份自己项目的"地图"——列出所有文件，每个文件用一句话说明作用，提交到 Git 仓库（课上教 git init/add/commit/push 最小流程）

---

#### 第 2 次：Git — 代码的存档与协作

> 解决：改崩了不知道怎么回退，多人写代码互相覆盖

**主线内容**
- 为什么需要版本管理（不是"文件夹改名备份"）
- 核心概念只讲一条线：工作区 → 暂存区 → 仓库
- 演示：add → commit → push → 在 GitHub 上看记录 → 改崩了用 git log + checkout 回退
- 分支概念（简单提）：为什么不直接改 main

**LLM 知识点：Context Window**
- 什么是上下文窗口，为什么聊久了 AI 开始"忘事"
- 怎么应对：新开对话、给 AI 喂关键上下文、用文档/PRD 代替口头描述
- 演示：同一个问题，在长对话末尾问 vs 新对话里带上下文问，效果对比

**动手**：把自己的项目推到 GitHub，做两次修改、两次 commit，在 GitHub 上看到提交记录

**作业**：故意改坏项目里一个功能，用 Git 回退到能跑的版本，截图提交。从这次开始，所有作业都通过 Git 提交

---

#### 第 3 次：读代码 — 看懂 AI 写了什么

> 解决：AI 生成了 200 行代码，不知道从哪看起

**主线内容**
- 读代码的方法：不是逐行看，而是"找入口 → 顺着数据流走"
- 拆解一个 React 项目的结构：入口文件 → 路由 → 页面组件 → 子组件 → 数据从哪来
- 核心 JS 概念速览（只讲"读代码时会遇到的"，不展开语法）：变量/函数/箭头函数、import/export、对象和数组、async/await
- 现场演示：拿一段 AI 生成的代码，从头到尾读一遍，边读边说思路

**LLM 知识点：Prompt 的结构化**
- 为什么"帮我做个页面"和"帮我做一个 React 页面，包含 XX 组件，数据从 /api/xx 接口获取，用 TypeScript"效果差很多
- 结构化 prompt 的要素：角色、上下文、具体需求、约束条件、输出格式
- 演示对比：同一个需求，随便说 vs 结构化写，生成的代码质量差异

**动手**：拿 AI 生成的一段代码，用自己的话写注释，描述每一块在做什么

**作业**：找自己项目里最核心的文件，写一段 50 字以内的概括："这个文件的作用是 XX，它从 XX 拿数据，渲染了 XX"

---

#### 第 4 次：React 心智模型 — 组件、状态、渲染

> 解决：AI 写了一堆 useState/useEffect，不知道为什么要这样写

**主线内容**
- 组件是什么：一个函数，输入 props，输出 UI
- 状态是什么：useState — 数据变了，页面自动更新（对比"手动改 DOM"）
- 渲染流程：状态改变 → 组件重新执行 → 返回新的 UI → 浏览器更新
- 不讲语法细节，重点讲心智模型：为什么 React 要这样设计
- 简单提 useEffect：什么时候用、为什么 AI 总爱生成它

**LLM 知识点：AI 的"幻觉"和过时知识**
- AI 可能用已经废弃的 API 或不存在的库
- 怎么判断 AI 生成的代码靠不靠谱：看 import 的库是否存在、API 文档是否匹配、跑一遍看报错
- 实用习惯：让 AI 标注它用了哪些库和版本，自己去官方文档验证

**动手**：在自己项目里找到一个 useState，改变初始值观察页面变化；加一个新状态控制元素的显示/隐藏

**作业**：画一张图（手绘/截图标注），描述项目某个页面的"组件树"和"数据从哪来到哪去"

---

#### 第 5 次：后端 — 数据存在哪、API 是什么

> 解决：AI 写了 fetch('/api/xxx')，不知道这行代码背后发生了什么

**主线内容**
- 前端和后端的分工：前端负责展示，后端负责数据和逻辑
- API 是什么：前后端之间的"约定"— 你给我什么参数、我还你什么数据
- Live coding：用 Node/Express 写一个最简单的接口（返回 JSON），前端 fetch 调用它
- GET vs POST 的区别
- 数据库概念：数据存文件 vs 存数据库，SQL 一句话解释

**LLM 知识点：怎么写需求文档给 AI**
- 当项目变复杂，一句话 prompt 不够了，需要写简单的 PRD
- PRD 的最小结构：做什么功能、数据从哪来、接口长什么样、页面交互逻辑
- 演示：用一份简单 PRD 让 AI 生成前后端代码，对比"随便说一句"的效果

**动手**：本地启动一个后端服务，用浏览器直接访问接口地址看到返回的 JSON；前端调通这个接口

**作业**：给自己项目加一个后端接口，实现"数据刷新不丢失"

---

#### 第 6 次：调试 + 部署 — 出了问题怎么办，做完了怎么上线

> 解决：报错看不懂只会反复问 AI；做完了只能 localhost 自己看

**调试部分**
- 报错信息怎么读：从最后一行往上看、关键词定位
- 常见错误模式：网络请求失败、变量 undefined、组件没渲染
- 演示：故意制造 3 个常见 bug，现场定位和修复
- 什么时候该自己调、什么时候该问 AI：判断标准

**部署部分**
- 用 Vercel 一键部署前端项目（从 GitHub 仓库直接部署）
- 部署完分享链接给别人用 — 这是"做完了"的真正标志

**LLM 知识点：怎么让 AI 帮你调试**
- 把报错信息完整复制给 AI（不要只截一行）
- 给上下文：告诉 AI 你在做什么、期望什么结果、实际什么结果
- AI 调试的局限：它看不到你的运行环境，有时候会"猜"一个修复方案反而改出新 bug
- 实用技巧：让 AI 先解释报错含义，再给修复方案，分两步走

**动手**：修复一段预先埋好 bug 的代码（不准直接丢给 AI）；把自己项目部署到 Vercel

**作业**：把部署后的链接发到群里，互相体验对方的项目

---

### 第 6 次之后：分头阶段

- **熊**：选一个具体项目（比如管理后台），定期 review 代码、给反馈
- **小苗**：走动效路线，推荐 Framer Motion / CSS Animation 资料，重点解决"怎么参与团队项目"的实操问题（分支、PR、Code Review 流程）
- **庞滚滚**：给一个完整独立项目目标（比如 Chrome 插件），自己推进，卡住了再找人

### 教学技巧

- **收住发散，逼出卡点**：发散问题先记到共享文档，课末统一处理；动手环节要求屏幕共享或截图，没发的就是卡住了，主动去问
- 每个知识点讲两层 — 第一层"是什么、怎么用"（所有人），第二层"为什么这样设计"（基础好的人多吸收）
- 录屏留存，方便回看，不需要做 PPT
- 鼓励先自己想一遍再看 AI 的答案
- 建共享问题库（飞书文档），遇到问题先记录，定期集中答疑

## 四、课前准备清单

### 你需要准备的

1. **一个 Demo 项目** — 不用从零写，找一个团队现有的、结构简单的真实页面。要求有前端（React）、有后端接口调用、代码量不大（一个文件能看完核心逻辑）。分享时所有演示都围绕这一个项目讲，避免每次换例子导致认知断裂。
2. **环境搭建指南** — 非技术同学卡在环境配置上的时间可能比学代码还长。提前写一份文档（Node.js 安装、VS Code + 推荐插件、Git 配置、项目怎么 clone 下来跑起来），在第一次分享**之前**就发给他们，搞不定的单独帮一下。
3. **每次分享后的练习题** — 不用复杂，一句话就行。目的是逼动手，不动手听完就忘。

### 不需要准备的

PPT。直接开编辑器 + 浏览器 live coding，录屏留存比 PPT 有用得多。

## 五、沟通话术：跟每个人怎么说

> 核心原则：不要一上来就发计划表，先让他们认同"为什么这样安排"，再说"怎么安排"。

### 跟熊说

**背景**：什么都想学，打了一堆 5 分，想 1-2 周搞定。需要帮她理解"先窄后宽"比"全面铺开"快。

> 你的热情特别好，但 1-2 周全学完不现实。最有效的路径是：先花 1-2 周**完整跑通一个小项目**（从前端到后端到上线），把全链路走一遍。这之后你再学任何单点知识都会快很多，因为你知道它在整个系统里的位置了。
>
> 所以前六周我们一起上课打基础，之后你挑一个想做的项目（比如你说的管理后台），我帮你 review 代码、答疑。这样其实比同时学十样东西更快到达"能独立做项目"的程度。

### 跟小苗说

**背景**：兴趣两极分化，目标是参与团队开发，但对一些基础内容打了 1 分。需要解释"集中课里有些内容你觉得不需要，但对你的目标其实是必要的"。

> 你的目标是能参与团队项目，这个目标其实比"自己做个小工具"要求更高——因为你需要能读懂别人的代码、能用 Git 协作、能理解项目的整体结构。
>
> 前六次集中分享的内容是帮你建立全局认知，哪怕有些点（比如后端）你现在觉得用不上，但理解它对你读懂团队项目的代码是有帮助的。集中课之后，我们会单独帮你搞定动效方向和团队协作流程。

### 跟庞滚滚说

**背景**：基础最好，节奏也最稳，需求最明确（补后端）。最好沟通，直接说就行。

> 你的前端基础已经不错了，前六次集中分享里前端部分你可能会觉得简单，但后端和全链路的部分应该正好是你需要的。集中课结束后，建议你直接开始做一个完整项目（比如你想做的 Chrome 插件），过程中卡住了随时找我。

### 确认时间的话术

不要在群里问"大家什么时候有空"，会来回扯很久。直接给一个具体提案：

> 我计划每周 X 下午 X 点，连续六周，每次 1 小时。如果这个时间有冲突跟我说，我调整。
