第 4 课教案：后端 — 数据存在哪、API 是什么

目标：让学员理解"前端和后端的分工"、"API 是前后端之间的约定"，并看懂把 Issue 看板从"假数据"升级为"从后端拿数据"的完整过程。

设计思路

第 3 课学员做了一个 Issue 看板，数据写死在 useState 里。这节课的核心动作是：把数据从前端搬到后端。

整节课围绕一个故事线：Issue 看板的数据搬家。数据只存在 JavaScript 变量里，不讲数据库、不讲 ORM、不讲 Redis。

先建立直觉——为什么要有后端：

想象你们用的 Meego。你在上面改了一个 Bug 的状态，团队里所有人刷新页面都能看到——不是因为你的电脑跟每个人的电脑直接连着，而是改动先发到 Meego 的服务器，服务器再返给所有人。如果没有服务器，你只能把数据用 U 盘拷给别人。

这就是后端存在的意义：互联网上有无数个互不认识的客户端，需要一个"所有人都连着的中间人"来存数据、同步状态、定规则。

0:07-0:14 — 概念：前端和后端各管什么 （7 分钟）

大家都用过 Meego——Meego 就是最典型的前后端分离应用。

前端 = Meego 客户端：Bug 列表长什么样、状态按钮在哪——你看到的一切、你的操作，都是前端。

后端 = Meego 服务器：所有 Bug 数据存在哪、谁负责哪个模块——数据存在哪、业务逻辑怎么算，都是后端。

API = 客户端和服务器之间的通信协议：你点"已完成"，客户端按什么格式打包、发到哪个地址、服务器怎么回——前端和后端之间的通信规则。

板书：

┌─────────────────┐                        ┌─────────────────┐
│ 前端（Meego客户端）│     API（通信协议）      │ 后端（Meego服务器）│
│                 │                        │                 │
│  显示页面        │ ──── 请求 ──────→      │  存数据          │
│  响应用户操作    │                        │  处理业务逻辑     │
│  更新界面        │ ←─── 响应（数据）──     │  返回结果         │
└─────────────────┘                        └─────────────────┘

0:18-0:22 — API 到底长什么样 （4 分钟）

动作：在浏览器地址栏输入 http://localhost:3001/api/issues，回车。

看到了吗？你刚才做的事情，就是手动调了一次 API。跟输入 baidu.com 没有本质区别——都是浏览器去找服务器要东西。区别只是 baidu.com 返回网页，这个地址返回纯数据。

API 要素Meego 里的例子代码里的例子
路径（URL）你在操作哪个资源/api/issues
方法（Method）你要做什么操作GET（查）、POST（新建）、PUT（改）、DELETE（删）
数据（JSON）改成什么状态{ "status": "已完成" }

0:22-0:35 — 看代码：后端长什么样 （13 分钟）

打开 server/index.ts，逐块讲解：

第一块：启动服务

import express from "express";
const app = express();
app.use(express.json());

Node.js 把 JavaScript 从浏览器解放出来，让它能跑在服务器上。Express 是 Node.js 上最流行的后端框架。

顺便说——后端用什么语言写很自由：Go、Java、Python 都可以。我们公司非联的后端大部分用 Go 写的。今天用 Node.js 是因为你们已经会 JavaScript 了。

这几行是固定写法，不需要记。

第二块：数据

const issues = [
  { id: 1, label: "Bug", title: "首页加载速度太慢", assignee: "小明", status: "进行中" },
  ...
];
👉 cue 学员：「这段数据你们之前在哪个文件里见过？」→ 等回答（App.tsx / useState 里）

对，这就是第 3 课 App.tsx 里 useState 那段数据，搬到了后端。多了一个 id 字段——每条 Issue 的身份证号，不会因为删除或排序而改变。

数据存在 JavaScript 变量里，跑在服务器内存中。刷新页面不会丢（刷新只重新加载前端）。

⚡ 预判学员问题：「那服务器重启呢？」→ 数据库你可以理解为服务器的硬盘——内存断电就没了，但硬盘断电还在。真实项目用数据库，今天先用变量把流程跑通。

第三块：GET API

app.get("/api/issues", (req, res) => {
  res.json(issues);
});
app .get ( "/api/issues" ,  (req, res) => { ... } )

 │    └── 方法：GET = 拿数据

第四块：PUT API（发速查卡！）

⚡ 操作提示：先把下面的速查卡发到群里，让学员看懂语法再看代码。
发给学员的速查卡：

① 对象：字节跳动 → 火山引擎 → 基础设施部门 → 负责人：张三
代码里 req.params.id 就是：req → params → id，用 . 一层层往里取。

② 数组：Meego 里的 Bug 列表就是数组，一条一条排着。issues.find(...) 就是逐条找。

③ 解构赋值：const { status } = req.body 等同于 const status = req.body.status，只是简写。

💡 看不懂的代码？选中它，Cursor 里按 Cmd+L 直接问。 
app.put("/api/issues/:id", (req, res) => {
  const id = Number(req.params.id);
  const { status } = req.body;
  const issue = issues.find(item => item.id === id);
  if (!issue) {
    res.status(404).json({ error: "Issue 不存在" });
    return;
  }
  issue.status = status;
  res.json(issue);
});

0:35-0:45 — Live Coding：前端对接后端 （10 分钟）

改动 1：useEffect 从后端拿数据（0:35-0:39）

打开 App.tsx，把写死的数据改成：

import { useState, useEffect } from 'react';

const [issues, setIssues] = useState([]);

useEffect(() => {
  fetch("/api/issues")
    .then(res => res.json())
    .then(data => setIssues(data));
}, []);

fetch 就是前端调 API 的方式——之前你在地址栏手动输入 URL，现在代码自动做了。state 一变，页面自动更新。

保存后打开 Network 面板验证：能看到 /api/issues GET 请求，状态 200。

改动 2：onChangeStatus 通知后端（0:39-0:45）

👉 cue 学员：「现在点'已完成'按钮再刷新，状态还是会回去，为什么？」→ 因为只改了前端 state，没告诉后端
⚡ 讲师操作提示：用 Cursor 直接改，不要现场手打
在 Cursor 里选中当前的 onChangeStatus 整块，按 Cmd+L，发这条提示词：

把这个 onChangeStatus 改成：先用 fetch PUT /api/issues/${issue.id} 告诉后端新状态，成功后再 fetch GET /api/issues 重新拿完整列表，然后 setIssues(data) 更新页面。用 method PUT，headers Content-Type application/json，body JSON.stringify({ status: newStatus })

AI 改完后在屏幕上讲解代码即可。

改完后的代码：

onChangeStatus={(newStatus) => {
  fetch(`/api/issues/${issue.id}`, {
    method: "PUT",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ status: newStatus }),
  })
    .then(() => fetch("/api/issues"))
    .then(res => res.json())
    .then(data => setIssues(data));
}}

关键测试：点"已完成"，然后刷新页面——状态还在！对比课开头，同样是刷新，现在数据不丢了。

0:45-0:53 — 演示：新增 Issue 的 API （8 分钟）

直接展示完整的 POST API 代码并讲解：

app.post("/api/issues", (req, res) => {
  const { label, title, assignee } = req.body;
  const newIssue = {
    id: issues.length + 1,
    label, title, assignee,
    status: "待处理",
  };
  issues.push(newIssue);
  res.json(newIssue);
});

跟 GET 和 PUT 的结构一模一样：app.方法(路径, 处理函数)。这三种 API（GET/POST/PUT）覆盖了"拿、增、改"三种操作。DELETE（删）你们作业里会用到。

0:53-0:56 — 总结 （3 分钟）

1. 前端管显示，后端管数据——它们通过 API 通信。

2. API = 路径 + 方法 + 数据——GET 拿、POST 加、PUT 改、DELETE 删。

3. 前端用 fetch 调 API——跟你在地址栏输入 URL 本质一样，代码自动做了。

速查卡（发群里）

GET → 查数据 → app.get("/路径", (req, res) => { ... })
POST → 新建 → app.post(...)
PUT → 改数据 → app.put(...)
DELETE → 删数据 → app.delete(...)

前端调 API：fetch("/api/xxx") → GET；fetch("/api/xxx", { method, headers, body }) → POST/PUT/DELETE

后端收数据：req.params.id（路径参数）、req.body（JSON 数据）
后端发数据：res.json(数据)、res.status(404).json(...)

关键原则

  1. 一条故事线：Issue 看板的数据搬家，不讲数据库、ORM、Redis
  2. 先看效果再看代码：每个 API 先在浏览器访问看到 JSON，建立直觉后再看代码
  3. cue 学员而不是让学员动手：每讲一块就 cue 学员回答问题
  4. Network 面板是贯穿线索：每次改完代码都打开 Network 面板看一眼
  5. 不讲底层：不解释 HTTP 协议、Promise 原理、Express 中间件机制
  6. 刷新是关键测试：全课两次刷新对比——开头丢了，结尾还在
  7. 用 Cursor AI 辅助 live coding：onChangeStatus 那块选中代码 Cmd+L 让 AI 改，降低现场出错风险
