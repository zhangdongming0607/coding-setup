第 4 课教案：后端 — 数据存在哪、API 是什么

目标：让学员理解"前端和后端的分工"、"API 是前后端之间的约定"，并亲手把第 3 课的 Issue 看板从"假数据"升级为"从后端拿数据"。

设计思路

第 3 课学员做了一个 Issue 看板，数据写死在 useState 里。这节课的核心动作是：把数据从前端搬到后端。

这个过程自然引出三个问题：

  1. 数据为什么不能一直写在前端？（动机）
  2. 后端怎么把数据给前端？（API）
  3. 前端怎么拿到后端的数据？（fetch）

整节课围绕一个故事线：Issue 看板的数据搬家。数据只存在 JavaScript 变量里，不讲数据库、不讲 ORM、不讲 Redis。

  • VS Code 打开项目，折叠 node_modules

课程流程

0:00-0:07 开场：数据写死的问题

目的：从第 3 课的 Issue 看板切入，让学员自己发现"数据写死"的问题，并连接到他们使用 AI 的真实体验。

动作：

  1. 打开浏览器，访问第 3 课的 Issue 看板（localhost:5173）
  2. 点几下状态按钮，把"登录功能异常"改成"已完成"
  3. 刷新页面

上节课我们做了这个 Issue 看板。大家看，我现在把"登录功能异常"改成"已完成"了。

现在我刷新一下页面——

（刷新）

没了。又变回"待处理"了。

为什么？我们打开代码看看。

打开 App.tsx，指向第 8-14 行：

const [issues, setIssues] = useState([
  {label: 'Bug', title: '首页加载速度太慢', assignee: '小明', status: '进行中'},
  {label: 'Bug', title: '登录功能异常', assignee: '小明', status: '待处理'},
  // ...
])

数据写死在代码里了。每次刷新页面，React 重新执行这个函数，useState 的初始值又是这个固定的数组。你改的那些状态，只存在页面的"临时记忆"（state）里，刷新就没了。

这就像你在白板上写了一串数字，擦掉白板重新画，数字当然没了。

连接 AI 使用场景：

你们平时让 AI 生成项目的时候，有没有遇到过这种情况：AI 帮你写了后端代码，但跑起来报错了，AI 跟你说"改一下 routes/api.js 第 15 行"，你打开一看完全看不懂——不知道那个文件在干嘛，也不知道怎么改。

今天这节课就是让你看得懂那些代码。我们要把数据从前端搬到一个独立的地方——后端。搬完之后，你就知道 AI 生成的后端代码到底在干嘛了。

0:07-0:14 概念：前端和后端各管什么

目的：用餐厅类比建立"前后端分工"的心智模型。

在讲怎么做之前，先搞清楚一个概念：前端和后端到底怎么分工。

第 1 课我们用餐厅类比讲过请求链路——浏览器发请求、服务器返数据。今天我们进厨房看看里面怎么运作的。

前端 = 餐厅大厅：菜单长什么样？桌椅怎么摆？客人怎么点菜？——你看到的一切、你的操作，都是前端

后端 = 厨房 + 仓库：食材存在哪？今天还有没有番茄？菜怎么做？——数据存在哪、业务逻辑怎么算，都是后端

API = 传菜窗口：大厅和厨房之间的窗口。服务员把点菜单递进去，厨房把做好的菜递出来——前端和后端之间的约定

板书（白板或飞书文档）：

┌─────────────────┐                        ┌─────────────────┐
│   前端（大厅）    │      API（窗口）        │   后端（厨房）    │
│                 │                        │                 │
│  显示页面        │ ──── 请求 ──────→       │  存数据          │
│  响应用户操作     │                        │  处理业务逻辑     │
│  更新界面        │ ←─── 响应（数据）──      │  返回结果         │
└─────────────────┘                        └─────────────────┘

回忆一下第 1 课的聊天 App——你在输入框打字、点发送，前端把消息通过 API 发给后端，后端返回回复，前端再把回复画到页面上。当时我们在 Network 面板里看到的那些请求，就是 API 调用。

今天要做的事情一样：让 Issue 看板的前端通过 API 去后端拿数据，而不是自己写死。

前端和后端是两个独立的程序，要分别启动。就像餐厅的大厅和厨房——大厅开门了不代表厨房也在运转，两边都得有人。

我们要开两个终端窗口：一个模拟大厅（前端），一个模拟厨房（后端）。

演示步骤：在 VS Code 中打开终端（Ctrl+`），点 + 号新开标签页。

终端角色命令端口
终端 1大厅（前端）cd lessons/lesson-4 && npm run devlocalhost:5173
终端 2厨房（后端）cd lessons/lesson-4/server && npm run devlocalhost:3001

两个端口号不一样——5173 是大厅的门牌号，3001 是厨房的门牌号。前端跑在 5173，后端跑在 3001，它们通过 API 互相通信。

0:18-0:22 API 到底长什么样

目的：让学员直观看到 API 的输入和输出，理解"路径 + 方法 + JSON"三要素。

API 听起来很抽象，但我现在让你们看一个活的。

动作：打开浏览器，在地址栏输入 http://localhost:3001/api/issues，回车。

看到了吗？浏览器显示了一段数据。这就是后端返回的 Issue 列表。

你刚才做的事情，就是手动调了一次 API。跟你平时在浏览器里输入 baidu.com 没有本质区别——都是浏览器去找一个服务器要东西。区别只是 baidu.com 返回的是一个网页，而这个地址返回的是纯数据。

[
  { "id": 1, "label": "Bug", "title": "首页加载速度太慢", "assignee": "小明", "status": "进行中" },
  { "id": 2, "label": "Bug", "title": "登录功能异常", "assignee": "小明", "status": "待处理" }
]

这个数据格式叫 JSON。你们其实已经见过了——它长得跟第 3 课 useState 里写的那些数据一模一样。

每个 API 有三个要素：

API 要素点菜类比例子
路径（URL）你要的是哪道菜/api/issues
方法（Method）你要干嘛GET（看菜单）、POST（下新单）、PUT（改单）
数据（JSON）菜单上写的内容{ "title": "...", "status": "..." }

等一下我们会看到，前端代码里用 fetch 函数做的事情，跟你刚才在地址栏输入 URL 做的事情，本质上是一样的——都是去后端请求数据。只不过 fetch 是代码自动做的。

0:22-0:35 看代码：后端长什么样

目的：打开后端代码，逐块讲解。每讲一块就 cue 学员互动，保持注意力。

刚才我们在浏览器里看到了 API 返回的数据。现在我们去看看后端代码——那些数据是哪里来的？

打开 server/index.ts，逐块讲解：

第一块：开店准备

import express from "express";

const app = express();
app.use(express.json());

两件事：引入 Express（"餐厅管理系统"），创建一个 app，告诉它"前端发来 JSON 数据帮我自动解析"。

这几行是固定写法，每个 Express 后端都长这样，不需要记。

第二块：数据

const issues = [
  { id: 1, label: "Bug", title: "首页加载速度太慢", assignee: "小明", status: "进行中" },
  { id: 2, label: "Bug", title: "登录功能异常", assignee: "小明", status: "待处理" },
  // ...
];

cue 学员：「这段数据你们之前在哪个文件里见过？」→ 等回答（App.tsx / useState 里）

对，这就是第 3 课 App.tsx 里 useState 那段数据，搬到了后端。

多了一个 id 字段——每条 Issue 的身份证号。为什么不用 index？因为删了第 2 条，后面每条的 index 都变了。id 不会因为删除或排序而改变。

数据存在 JavaScript 变量里，跑在服务器内存中。刷新页面不会丢——因为刷新只重新加载前端，后端变量不受影响。服务器重启会丢——真实项目用数据库，今天先用变量把流程跑通。

第三块：API — 获取所有 Issue

// GET /api/issues —— 获取所有 Issue
app.get("/api/issues", (req, res) => {
  res.json(issues);
});

在白板上拆解：

app .get ( "/api/issues" ,  (req, res) => { ... } )
 │    │         │                │
 │    │         │                └── 收到请求后做什么（函数）
 │    │         └── 路径（就是浏览器地址栏输入的那个）
 │    └── 方法：GET = 拿数据
 └── 你的餐厅 app

翻译成人话：当有人用 GET 方法访问 /api/issues 时，把 issues 数组以 JSON 格式返回。

req = request，前端发过来的东西。res = response，你要回给前端的东西。

这三行代码，就是一个完整的 API。

cue 学员验证：「大家回忆一下，刚才在浏览器里访问 localhost:3001/api/issues 看到了什么？那段 JSON 就是这三行代码返回的。能对上吧？」

第四块：API — 修改 Issue 状态

// PUT /api/issues/:id —— 修改一个 Issue 的状态
app.put("/api/issues/:id", (req, res) => {
  const id = Number(req.params.id);
  const { status } = req.body;

  const issue = issues.find(item => item.id === id);
  if (!issue) {
    res.status(404).json({ error: "Issue 不存在" });
    return;
  }

  issue.status = status;
  res.json(issue);
});

cue 学员：「app.put 的 put 是什么意思？GET 是拿数据，那 PUT 呢？」→ 等回答（改数据）

逐行翻译：

  1. "/api/issues/:id" — :id 是占位符。请求 /api/issues/3 时，req.params.id 就是 3。就像跟服务员说"3 号桌"
  2. req.body — 前端发过来的数据（"要改成什么状态"）
  3. issues.find(...) — 在数组里找到对应的 Issue
  4. 找不到返回 404——第 1 课讲的，4 开头是"你的问题"
  5. 找到了就改 status，把改完的结果返回

cue 学员：「用餐厅类比的话，这个 API 在做什么？」→ 期望回答方向：客人跟服务员说"3 号桌改成微辣"，服务员找到单子，改好，回来说"好了"

第五块：开门营业

app.listen(3001, () => {
  console.log("后端跑在 http://localhost:3001");
});

让服务器监听 3001 端口——厨房的门牌号是 3001，前端通过这个门牌号来找后端。

cue 学员总结：「整个后端代码做了几件事？谁来总结一下？」→ 期望回答：开店、定菜单、开门

后端代码比前端简单——它不管页面长什么样，只管"你要什么数据，我给你什么数据"。

0:35-0:45 Live Coding：前端对接后端

目的：现场修改前端代码，让 Issue 看板从后端拿数据。全课核心环节。

后端准备好了，API 也测过了。现在来改前端——让 Issue 看板不再用假数据，而是从后端拿真数据。

改动 1：页面加载时从后端获取数据（0:35-0:39）

当前代码（写死）→ 改成从后端获取：

import { useState, useEffect } from 'react';

// 改成空数组——一开始没有数据
const [issues, setIssues] = useState([]);

// 页面加载后，去后端拿数据
useEffect(() => {
  fetch("/api/issues")
    .then(res => res.json())
    .then(data => setIssues(data));
}, []);

三行 fetch，三步操作：

  1. fetch("/api/issues") — 去后端拿数据（就是刚才在浏览器里访问的那个地址）
  2. .then(res => res.json()) — 把返回的 JSON 解析出来
  3. .then(data => setIssues(data)) — 放到 state 里 → 页面自动更新

fetch 就是前端调 API 的方式——之前你在浏览器地址栏手动输入 URL，现在代码自动做了。

顺便说一句 Vite 代理：

你们可能注意到了：代码里写的是 fetch("/api/issues")，不是 fetch("http://localhost:3001/api/issues")。因为项目配了一个转发规则——前端看到 /api 开头的请求，自动帮你转发到后端 3001 端口。就像大厅有个内部通道直通厨房，不用出门绕后门。

演示效果：保存文件，切到浏览器。卡片出来了。打开 Network 面板 → 刷新 → 能看到 /api/issues 请求，GET 方法，状态 200。

代码里的 fetch → Network 面板里的请求 → 后端的 app.get → 返回数据 → 前端 setIssues → 页面更新。整条链路通了。

改动 2：修改状态时通知后端（0:39-0:45）

cue 学员：「现在点'已完成'再刷新，状态还是会回去。谁来说说为什么？」→ 等回答（只改了前端 state，没告诉后端）

改成先告诉后端，再重新拿数据：

onChangeStatus={(newStatus) => {
  fetch(`/api/issues/${issue.id}`, {
    method: "PUT",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ status: newStatus }),
  })
    .then(() => fetch("/api/issues"))
    .then(res => res.json())
    .then(data => setIssues(data));
}}

跟刚才的 fetch 比，多了几样东西：

  • method: "PUT" — 不再是默认的 GET，而是 PUT（改数据）
  • `/api/issues/${issue.id}` — 路径带 id，告诉后端"改第几条"
  • headers — 告诉后端"发的是 JSON 格式"
  • body: JSON.stringify({ status: newStatus }) — 要改成什么状态

后端改完之后，前端重新 fetch("/api/issues") 拿最新列表——跟页面加载时拿数据是一模一样的模式。

演示效果：

  1. 点"已完成" → Network 面板多了 PUT 请求 + GET 请求
  2. 关键测试——刷新页面 → 状态还在！

对比课开头：同样是刷新，开头数据丢了，现在数据还在。区别就在于——数据存在后端了。

0:45-0:53 动手环节：补全"新增 Issue"的 API

目的：让学员自己补全一个 API，巩固理解。用填空题降低门槛。

现在能拿数据、能改状态了。但看板还缺一个功能——添加新 Issue。这个交给你们来写。

我已经把代码骨架写好了，你们需要填空——把几个关键的地方补上。

填空题（发到群里）：

// POST /api/issues —— 新增一个 Issue
app.post("/api/issues", (req, res) => {
  // 第 1 步：从前端发来的数据里取出 label、title、assignee
  const { label, title, assignee } = ________;

  // 第 2 步：组装一条新 Issue
  const newIssue = {
    id: issues.________ + 1,
    label,
    title,
    assignee,
    status: "________",
  };

  // 第 3 步：加到数组里
  issues.________(newIssue);

  // 第 4 步：返回给前端
  res.________(newIssue);
});

提示：

  • 第 1 步：前端发来的数据存在哪个对象里？（PUT 那个 API 里用过）
  • 第 2 步：数组有几个元素用什么属性？新 Issue 默认什么状态？
  • 第 3 步：往数组末尾加元素的方法？
  • 第 4 步：后端返回 JSON 用什么方法？（GET 那个 API 里用过）

验证方式：打开浏览器 DevTools → Console 面板，粘贴代码：

fetch("/api/issues", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    label: "需求",
    title: "加一个搜索功能",
    assignee: "小红"
  })
}).then(r => r.json()).then(console.log)

讲师动作：给学员 5 分钟独立尝试，鼓励互相讨论，5 分钟后统一对答案。

常见卡点：

卡点应对
第 1 步不知道填什么"前端发来的数据存在 req 的哪个属性里？讲 PUT 的时候说过——"
第 2 步不知道怎么获取数组长度"一个数组有几个元素，用什么属性？就像问'这个班有多少个学生'"
第 3 步不知道用什么方法"JavaScript 里往数组末尾加东西的方法，名字叫 push"
新 Issue 加了但刷新又没了后端被 tsx watch 重启了，数据丢了——好教学时刻："这就是为什么真实项目要用数据库"

答案：

app.post("/api/issues", (req, res) => {
  const { label, title, assignee } = req.body;        // 前端发来的数据

  const newIssue = {
    id: issues.length + 1,                              // 数组长度 + 1
    label,
    title,
    assignee,
    status: "待处理",                                    // 默认待处理
  };

  issues.push(newIssue);                                // 加到数组末尾
  res.json(newIssue);                                   // 返回 JSON
});

跟 GET 的结构一模一样：app.方法(路径, 处理函数)。GET 是给数据，POST 是收数据。

三种 API（GET / POST / PUT）覆盖了"拿、增、改"三种操作。真实项目里还有 DELETE（删），作业里会用到。

0:53-0:56 总结

今天的核心就三件事：

1. 前端管显示，后端管数据。前端决定页面长什么样，后端决定数据存在哪。通过 API 通信。

2. API = 路径 + 方法 + 数据。路径去哪找、方法做什么、JSON 传数据。

3. 前端用 fetch 调 API。fetch(url) 拿数据，fetch(url, { method, body }) 发数据。

学员的第 3 课项目丢了让他 git pull 拉最新代码，直接用 lessons/lesson-4/ 下的项目
后端启动报端口被占用lsof -i :3001 查谁占了，kill 掉；或者改成 3002 端口（同时改 Vite 代理）
前端请求后端报 CORS 错误检查 Vite 代理配置是否正确。代理配好就不会有 CORS 问题
学员对 fetch 三行代码完全懵不解释 Promise 原理。就说"这三行是固定模板——第一行是地址，第二行是解析，第三行是用数据。先照着用"
学员在填空环节完全卡住缩小范围——先只填第 3 步和第 4 步（push 和 json），这两个最直观
req.body 是 undefined 导致报错99% 是 app.use(express.json()) 没写或写在了 API 定义后面。帮学员把这行移到最前面
学员改后端代码后 tsx watch 重启，数据丢了这是预期行为。利用这个时刻解释"变量只在内存里，重启就没了，真实项目要用数据库"
课程进度比预期快让学员做加分任务（前端加表单），或现场演示用 Console 调 DELETE API
课程进度比预期慢砍掉动手环节，改成讲师演示填空 + 学员跟着在 Console 里验证。前端 onChangeStatus 改动可简化为只讲思路不现场改

关键原则

  1. 一条故事线：Issue 看板的数据搬家。所有内容围绕这件事展开，不讲数据库、不讲 ORM、不讲部署
  2. 延续第 3 课的项目：学员对 Issue 看板已经熟悉，在已知的项目上做增量修改，认知负担最小
  3. 先看效果再看代码：每个 API 先在浏览器里访问看到 JSON 数据，建立直觉后再看后端代码怎么写的
  4. cue 学员而不是让学员动手：后端代码讲解环节，每讲一块就 cue 学员回答问题，保持注意力但不引入操作风险
  5. Network 面板是贯穿线索：每次写完 API 或改完前端，都打开 Network 面板看一眼，强化"代码的效果是可观察的"。衔接第 1 课教的 DevTools 用法
  6. 不讲底层：不解释 HTTP 协议、不讲 Promise 原理、不讲 Express 中间件机制。fetch 三行代码当固定模板用
  7. 刷新是关键测试：全课两次"刷新页面"对比——开头丢了（引出问题），结尾还在（证明解决了）。前后呼应最有说服力
  8. 动手环节用填空题：不让学员从零写代码，给骨架让他们填关键部分，降低门槛

App.tsx（改完后的最终版本）

import { IssueCard } from './components/issue-card';
import { useState, useEffect } from 'react';

export default function App() {
  const [issues, setIssues] = useState([]);

  // 页面加载后，从后端拿数据
  useEffect(() => {
    fetch("/api/issues")
      .then(res => res.json())
      .then(data => setIssues(data));
  }, []);

  return (
    <div className="app">
      <h1>Issue 看板</h1>
      <p>从这里开始写第一个组件 👇</p>
      {issues.map((issue) => {
        return <IssueCard
          key={issue.id}
          assignee={issue.assignee}
          label={issue.label}
          title={issue.title}
          status={issue.status}
          onChangeStatus={(newStatus) => {
            fetch(`/api/issues/${issue.id}`, {
              method: "PUT",
              headers: { "Content-Type": "application/json" },
              body: JSON.stringify({ status: newStatus }),
            })
              .then(() => fetch("/api/issues"))
              .then(res => res.json())
              .then(data => setIssues(data));
          }}
        />
      })}
    </div>
  );
}

server/index.ts（含 POST 答案）

import express from "express";

const app = express();
app.use(express.json());

const issues = [
  { id: 1, label: "Bug", title: "首页加载速度太慢", assignee: "小明", status: "进行中" },
  { id: 2, label: "Bug", title: "登录功能异常", assignee: "小明", status: "待处理" },
  { id: 3, label: "Bug", title: "注册功能异常", assignee: "小明", status: "进行中" },
  { id: 4, label: "Bug", title: "忘记密码功能异常", assignee: "小明", status: "待处理" },
  { id: 5, label: "Bug", title: "用户中心功能异常", assignee: "小明", status: "进行中" },
];

// GET /api/issues
app.get("/api/issues", (req, res) => {
  res.json(issues);
});

// PUT /api/issues/:id
app.put("/api/issues/:id", (req, res) => {
  const id = Number(req.params.id);
  const { status } = req.body;
  const issue = issues.find((item) => item.id === id);
  if (!issue) {
    res.status(404).json({ error: "Issue 不存在" });
    return;
  }
  issue.status = status;
  res.json(issue);
});

// POST /api/issues（动手环节答案）
app.post("/api/issues", (req, res) => {
  const { label, title, assignee } = req.body;
  const newIssue = {
    id: issues.length + 1,
    label,
    title,
    assignee,
    status: "待处理",
  };
  issues.push(newIssue);
  res.json(newIssue);
});

app.listen(3001, () => {
  console.log("后端跑在 http://localhost:3001");
});
