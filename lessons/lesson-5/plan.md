# 第 6 课：调试 + 部署 — 出了问题怎么办，做完了怎么上线

> 目标：让学员掌握"读报错 → 定位问题"的调试方法论，并亲眼看到 Issue 看板从 localhost 部署到公网、任何人都能访问。

## 设计思路

第 4 课学员把 Issue 看板从"假数据"升级为"前后端通信"。现在项目能在本地跑通了，但还差最后两步：

1. **代码报错了怎么办？** 学员用 AI 写代码时最常遇到的困境不是"不会写"，而是"报错了看不懂"。这节课教一套可重复的流程：看 Console → 读第一行 → 找文件行号 / 看 Network → 读状态码 → 点开看详情。
2. **代码跑通了怎么让别人用？** 把项目从 localhost 部署到 Vercel，拿到公网 URL。

**重心分配：调试是主角（28 分钟），部署是配角（15 分钟）。**

调试是学员课后最高频需要的技能——每次 AI 生成的代码跑不起来，第一反应应该是"看报错"而不是"继续问 AI"。部署只需要做一次，且步骤固定，适合快速演示。

**调试环节的核心设计：诊断卡。** 讲师现场制造 3 个错误，投屏给学员看，学员在纸上（或飞书文档）填写"Console 说了什么、Network 状态码、可能原因"。这比纯看演示多了主动思考的环节，又避免了学员环境不一致的风险。

### 0:05-0:12 — 调试方法论：你该看哪里

**目的**：在动手修 bug 之前，先建立"去哪找线索"的意识。不讲太多理论，够用就行，后面马上用到。

**话术**：

> 在修之前，先说一个最基本的问题：**报错信息在哪看？**
>
> 两个地方：

**打开 Chrome DevTools（F12）**：

> **第一个：浏览器 Console 面板。**
> - 红色 = 错误，程序跑不下去了
> - 黄色 = 警告，还能跑但有隐患
> - 这里能看到前端 JavaScript 的报错
>
> **第二个：终端（跑后端的那个窗口）。**
> - 后端的报错不会出现在浏览器里，出现在终端里
> - 看到 `Error`、`Cannot`、`failed` 这种关键词就说明有问题
>
> **记住一句话：前端报错看 Console，后端报错看终端。**

**报错信息的结构**：

> 大部分报错长这样：

**板书**：

```
TypeError: Cannot read properties of undefined (reading 'map')
│                                                        │
└── 第一行：什么坏了                                       │
                                                         │
    at App (App.tsx:15:22)                               │
              │       └── 第 15 行第 22 个字符              │
              └── 文件名                                   │
```

> 只需要看两样东西：
> 1. **第一行**——告诉你"什么坏了"
> 2. **文件名 + 行号**——告诉你"去哪找"
>
> 好，理论够了。我们回到刚才的白屏——来看看它到底报了什么。

### 0:20-0:30 — Bug 2：点状态按钮没反应 → 打日志 → 喂给 AI

**目的**：这是本节课最重要的环节——教"打日志 → 观察 → 喂给 AI"的完整工作流。

**讲师操作**：

1. 点一张卡片的"已完成"按钮
2. 状态没变
3. 打开 Network 面板：PUT `/api/issues/1` 显示红色，状态 500 或返回异常数据

**话术**：

> 点了"已完成"，但状态没变。
>
> 看 Network——PUT 请求出问题了。但这次 Console 和 Network 给的信息都不太够，只知道"后端出错了"，不知道具体哪里出错。
>
> 这种时候怎么办？**打日志。**
>
> 什么是打日志？就是在代码里加 `console.log`，让程序运行到那一步时把数据打印出来给你看。就像破案的时候在嫌疑人身上装了个定位器——你能看到数据走到哪一步出了问题。

**打开 `server/index.ts`，在 PUT 处理函数里加日志**：

```typescript
app.put("/api/issues/:id", (req, res) => {
  console.log("===== 收到 PUT 请求 =====");     // ← 加这行
  console.log("请求参数 id:", req.params.id);     // ← 加这行
  console.log("请求 body:", req.body);            // ← 加这行

  const id = Number(req.params.id);
  const { status } = req.body;
  // ...
```

**保存（tsx watch 自动重启后端），再点一次"已完成"按钮。**

**切到后端终端**，指给学员看：

```
===== 收到 PUT 请求 =====
请求参数 id: 1
请求 body: undefined
```

**话术**：

> 看终端输出：
> - `请求参数 id: 1` —— 这个正常
> - `请求 body: undefined` —— **这就是问题所在**
>
> 前端发了 `{ status: "已完成" }` 过来，但后端收到的 `req.body` 是 `undefined`——数据没传过来。或者说，数据传过来了，但后端**不知道怎么解析**。
>
> 到这一步你已经知道了问题的关键信息：**req.body 是 undefined**。
>
> 接下来有两条路：你可以自己去搜，也可以**把这个线索交给 AI**。

**演示"把日志喂给 AI"的工作流**：

> 我在 Cursor 里按 `Cmd+L` 打开对话框，告诉 AI：

**投屏展示发给 AI 的内容**：

```
我的 Express 后端，PUT 请求的 req.body 是 undefined。
终端日志：
  请求参数 id: 1
  请求 body: undefined
前端发的是 JSON 格式的数据。请问是什么原因？
```

**话术**：

> 注意我发给 AI 的东西——不是一大段报错信息，而是**打日志之后观察到的关键线索**。这样 AI 能秒回。

**展示 AI 的回复**（预期 AI 会说：缺少 `express.json()` 中间件）：

> AI 说："你可能缺少了 `express.json()` 中间件。加上 `app.use(express.json())`。"
>
> 我们去看看——

**打开 `server/index.ts`，指向第 8 行**：

```typescript
// app.use(express.json());  // ← 被注释掉了！
```

> 果然，这行被注释掉了。Express 默认不解析 JSON——需要这行代码告诉它"如果前端发来的是 JSON，帮我自动解析"。没有它，`req.body` 就永远是 `undefined`。
>
> 取消注释：

```typescript
app.use(express.json());
```

**保存，后端自动重启，再点"已完成"——状态变了！**

> 好，第二个 bug 也修好了。
>
> 我们刚才做了什么？

**板书**：

```
调试三步走（遇到看不懂的报错时）：

1. 打日志  →  在关键位置加 console.log，看数据走到哪一步出了问题
2. 看输出  →  终端或 Console 里，找到那个"不对"的值
3. 喂给 AI →  把关键线索（不是整段报错）告诉 AI，让它帮你定位原因
```

> **关键是第 1 步和第 2 步。** 如果你直接把一大段报错扔给 AI，AI 可能给你一个泛泛的回答。但如果你先打日志，把问题缩小到"req.body 是 undefined"这一个线索，AI 能直接告诉你答案。
>
> **打日志不难——就是加 `console.log`。** 你不确定哪里出了问题，就在可疑的地方加一行，看它打出来什么。

**（给 1 分钟填诊断卡第 2 行）**

### 0:30-0:33 — 补充：服务端报错怎么看

**目的**：快速补充一个场景——后端代码本身报错（不是 req.body 的问题，而是代码写错导致 crash）。让学员知道"终端不只是看日志，也能看报错"。

**话术**：

> 刚才两个 bug 都是"代码能跑，但行为不对"。还有一种情况更直接——**后端代码本身就跑不起来，直接崩了。**
>
> 比如你在后端写了这样一行：

**在 `server/index.ts` 任意位置加一行故意写错的代码**（现场演示，不改源文件）：

```typescript
issues.forEach(item => item.name.toUpperCase());  // item 上没有 name 属性
```

**保存，看终端**：

```
TypeError: Cannot read properties of undefined (reading 'toUpperCase')
    at server/index.ts:20:35
```

> 看到了吗？终端直接打出了报错——`TypeError`，在 `server/index.ts` 第 20 行。
>
> 跟浏览器 Console 里的报错结构一模一样：**类型 + 描述 + 文件行号**。读法也一样：看第一行，找行号。
>
> 所以前端报错看 Console，后端报错看终端——**读法是一样的**。

**撤销刚才加的那行代码，恢复正常。**

> 记住：后端没反应的时候，**第一件事去看终端**。如果终端有红色报错，读法跟 Console 一样。

### 0:37-0:42 — 部署概念：从 localhost 到公网 URL

**目的**：用最简单的话解释"部署是什么"，为接下来的实操做铺垫。

**话术**：

> 调试讲完了。接下来讲第二件事——**部署**。
>
> 我们的 Issue 看板现在跑在 `localhost:5173`。`localhost` 就是"你自己的电脑"——只有你能打开这个地址，别人打开看到的是空白。
>
> **部署就是把你的代码放到别人的电脑上跑。** 这个"别人的电脑"是云服务器——一台 24 小时开机、接着互联网的电脑。代码放上去之后，它会给你一个公网 URL，任何人打开浏览器访问这个 URL，就能用你的应用。
>
> 类比：你在家做了一道菜（localhost），只有你自己能吃。你把菜送到一个餐厅（云服务器），餐厅给你一个地址，朋友去餐厅就能吃到了。
>
> 部署有很多种方式——买服务器自己配、用 Docker、用各种云平台。我们今天用 **Vercel**，原因很简单：**它是最傻瓜的方式**——连上 GitHub 仓库，它自动帮你构建、部署、给 URL。不需要你配置服务器。

**关于后端怎么办**：

> 我们的项目有前端也有后端。前端部署很简单——Vercel 帮你 `npm run build` 打包，把 HTML/CSS/JS 放到它的服务器上。
>
> 后端呢？在本地我们是单独跑了一个 Express 进程。但 Vercel 不是让你跑一个 24 小时在线的 Express 服务器——它用另一种方式：项目里有一个 `api/` 文件夹，里面放了一个函数文件，Vercel 会**自动把它变成后端接口**。
>
> 这个函数的写法跟 Express 略有不同，但原理完全一样——收到请求，返回数据。我已经提前写好了，你们有兴趣课后可以打开 `api/issues.ts` 看看。
>
> 今天的重点不是理解这个文件怎么写，而是看到**部署的完整流程**。

### 0:42-0:52 — 部署实操：Vercel 一键部署

**目的**：讲师全程直播部署过程，学员看到从 `git push` 到拿到公网 URL 的完整流程。

**话术**：

> 现在来实际操作。

#### 第一步：推代码到 GitHub（1 分钟）

**讲师操作**：

```bash
cd lessons/lesson-5
git add .
git commit -m "第 5 课项目代码"
git push
```

**话术**：

> 第一步——把代码推到 GitHub。Vercel 是从 GitHub 仓库读代码的，所以代码得先在 GitHub 上。
>
> 这个操作你们第 2 课学过——`git add`、`git commit`、`git push`，三步走。

#### 第二步：在 Vercel 上创建项目（3 分钟）

**讲师操作**：

1. 打开浏览器，访问 `vercel.com`，登录（用 GitHub 账号）
2. 点击 "Add New..." → "Project"
3. 在 GitHub 仓库列表中选择课程仓库
4. **关键设置**：Root Directory 填 `lessons/lesson-5`（因为代码不在仓库根目录）
5. Framework Preset：Vite（通常自动检测）
6. 点 "Deploy"

**话术**：

> 打开 Vercel 网站，用 GitHub 登录——一键就行。
>
> 然后"添加新项目"，选我们的 GitHub 仓库。
>
> **注意这里**——Root Directory 要设成 `lessons/lesson-5`。因为我们的仓库里有很多课的代码，要告诉 Vercel 只看第 5 课的文件夹。
>
> Framework 它自动检测到了 Vite——对的。
>
> 然后点 Deploy。就这么简单。

#### 第三步：等待构建（2 分钟）

**话术**：

> 现在 Vercel 在做的事情：
> 1. 从 GitHub 拉代码
> 2. 运行 `npm install`——安装依赖
> 3. 运行 `npm run build`——打包前端代码
> 4. 检测到 `api/` 文件夹——自动把里面的函数注册为后端接口
> 5. 把打包好的前端文件和后端函数放到它的服务器上

## 应急预案

| 情况 | 应对 |
|---|---|
| Vercel 部署失败（构建报错） | 直接打开提前部署好的备用 URL 演示效果。课后排查构建问题 |
| Vercel 网站打不开（网络问题） | 用手机热点试一次。如果还是不行，跳过实操，用备用 URL 的截图讲解部署结果，口述步骤 |
| 调试演示时出现意外错误 | 这反而是好教学素材——"看，我也遇到了一个没预期到的错误，我们用同样的方法来看它" |
| 场景 3（express.json）效果不明显 | 有时 `req.body` 是 undefined 不会报 500，而是返回奇怪的结果。引导学员看终端日志输出——"body 是 undefined，这就是线索" |
| 学员诊断卡全填错 | 不要紧，这正是学习的过程。对答案时慢慢讲解，强调"方法比答案重要" |
| 课程进度比预期快 | 加一个互动：让学员描述自己用 AI 时遇到的报错，现场分析一两个。或演示：改代码 → push → Vercel 自动重新部署 |
| 课程进度比预期慢 | 压缩部署概念环节（33-38）到 2 分钟，或砍掉六节课回顾改为一句话带过 |
| 后端启动报端口被占用 | `lsof -i :3001` 查谁占了端口，`kill` 掉；或改成 3002（同时改 Vite 代理配置） |

## 关键原则提醒

1. **调试是主角，部署是配角。** 如果时间不够，压缩部署而不是压缩调试。调试是学员课后每天都需要的技能。
2. **项目自带 bug，不需要现场制造。** 开场直接跑项目发现白屏，比"假设遇到报错"更有冲击力，也更可控。
3. **教方法，不教记忆。** 不要求学员记住每种错误码的含义。教的是一个可重复的流程：看 Console → 看 Network → 打日志 → 喂给 AI。
4. **"打日志 → 喂给 AI"是核心工作流。** 这是学员课后真正会用的调试方式——先自己缩小问题范围，再让 AI 精准回答。比直接把整段报错扔给 AI 高效得多。
5. **诊断卡是主动思考工具。** 目的不是考试，而是让学员从"看演示"变成"自己判断"。填错了完全没关系，对答案时才是教学的重点。
6. **部署不解释 Serverless 原理。** `api/issues.ts` 一笔带过——"已经写好了，Vercel 自动处理"。学员只需要知道部署的流程，不需要理解 Serverless 的工作机制。
7. **前端代码零改动是最好的教学点。** 从本地到部署，`src/` 里的代码一个字都不用改。这是最直观的"好架构"体现。
8. **呼应第一课。** 第一课教了 DevTools，今天用 DevTools 来调试；第一课讲了浏览器到服务器的链路，今天把项目部署到了服务器上。形成闭环。
9. **这是最后一课，收束感很重要。** 不是"下课了"，而是"你们已经有了完整的能力链路"。结尾给方向感，让学员知道接下来可以自己继续。
