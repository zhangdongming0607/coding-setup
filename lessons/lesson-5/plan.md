# 第 6 课：调试 + 部署 — 出了问题怎么办，做完了怎么上线

> 目标：让学员掌握"读报错 → 定位问题"的调试方法论，并亲眼看到 Issue 看板从 localhost 部署到公网、任何人都能访问。

## 设计思路

第 4 课学员把 Issue 看板从"假数据"升级为"前后端通信"。现在项目能在本地跑通了，但还差最后两步：

1. **代码报错了怎么办？** 学员用 AI 写代码时最常遇到的困境不是"不会写"，而是"报错了看不懂"。这节课教一套可重复的流程：看 Console → 读第一行 → 找文件行号 / 看 Network → 读状态码 → 点开看详情。
2. **代码跑通了怎么让别人用？** 把项目从 localhost 部署到 Vercel，拿到公网 URL。

**重心分配：调试是主角（28 分钟），部署是配角（15 分钟）。**

调试是学员课后最高频需要的技能——每次 AI 生成的代码跑不起来，第一反应应该是"看报错"而不是"继续问 AI"。部署只需要做一次，且步骤固定，适合快速演示。

**调试环节的核心设计：提问互动。** 讲师带着学员修两个预埋 bug，每一步用提问 cue 学员口头回答（"Console 说了什么？""状态码多少？"），保持注意力但不引入操作风险。

### 0:05-0:12 — 调试方法论：你该看哪里

**目的**：在动手修 bug 之前，先建立"去哪找线索"的意识。不讲太多理论，够用就行，后面马上用到。

**话术**：

> 在修之前，先说一个最基本的问题：**报错信息在哪看？**
>
> 两个地方：

**打开 Chrome DevTools（F12）**：

> **第一个：浏览器 Console 面板。**
> - 红色 = 错误，程序跑不下去了
> - 黄色 = 警告，还能跑但有隐患
> - 这里能看到前端 JavaScript 的报错
>
> **第二个：终端（跑后端的那个窗口）。**
> - 后端的报错不会出现在浏览器里，出现在终端里
> - 看到 `Error`、`Cannot`、`failed` 这种关键词就说明有问题
>
> **记住一句话：前端报错看 Console，后端报错看终端。**

**报错信息的结构**：

> 大部分报错长这样：

**板书**：

```
TypeError: Cannot read properties of undefined (reading 'map')
│                                                        │
└── 第一行：什么坏了                                       │
                                                         │
    at App (App.tsx:15:22)                               │
              │       └── 第 15 行第 22 个字符              │
              └── 文件名                                   │
```

> 只需要看两样东西：
> 1. **第一行**——告诉你"什么坏了"
> 2. **文件名 + 行号**——告诉你"去哪找"
>
> 好，理论够了。我们回到刚才的白屏——来看看它到底报了什么。

### 0:12-0:20 — Bug 1：页面白屏 → Network 404 → 路径写错了

**目的**：用刚讲的方法定位第一个 bug。每一步 cue 学员口头回答。

**讲师操作**：

1. 打开 Console 面板 → 显示红色报错 `加载失败`
2. 打开 Network 面板 → `/api/issue` 请求是红色，Status 显示 **404**

**话术**：

> 看 Console——有红色报错。

**（cue 学员）**：「Console 里那行红字写了什么？」

等学员回答（加载失败）。

> 对，"加载失败"。信息不多，但至少知道是加载数据出了问题。再看 Network 面板——

**（cue 学员）**：「`/api/issue` 那行是什么颜色？状态码是多少？」

等学员回答（红色，404）。

**（cue 学员）**：「404 是什么意思？第 1 课讲过。」

等学员回答（找不到 / 路径不对）。

> 对，"找不到这个地址"。后端没有 `/api/issue` 这个路径——仔细看，**少了一个 `s`**。后端定义的是 `/api/issues`。
>
> 我们去修。

**打开 `src/api/issues.ts`**，指向第 10 行：

```typescript
const res = await fetch('/api/issue');  // ← 少了个 s
```

**改成**：

```typescript
const res = await fetch('/api/issues');
```

**保存，页面自动刷新**——Issue 卡片出来了！

**话术**：

> 好了，第一个 bug 修好了。整个过程：
> 1. Console 看到"加载失败"
> 2. Network 看到 404
> 3. 404 = 路径不对 → 对比前后端路径 → 发现少了个 s → 改回来
>
> **404 基本都是路径拼写问题。** 但先别高兴——点一下状态按钮试试。

### 0:20-0:30 — Bug 2：点状态按钮报错 → 打日志 10 秒定位

**目的**：教"打日志"的习惯。重点不是这个 bug 有多难——而是**你自己看 10 秒比直接扔给 AI 更快**。打一行 `console.log`、看一眼输出，问题就定位了。

**预埋的 bug**：`server/index.ts` 第 8 行，`app.use(express.json())` 被注释掉了。Express 默认不解析 JSON，所以前端发来的数据后端读不到，`req.body` 永远是 `undefined`。

**讲师操作**：

1. 点一张卡片的"已完成"按钮
2. 状态没变
3. 打开 Network 面板：PUT `/api/issues/1` 显示红色，状态 500

**话术**：

> 好，卡片出来了。我点一下"已完成"——
>
> （点击）
>
> 没反应。看 Network——PUT 请求红了，500。
>
> 500 是"后端代码出错了"。但具体出了什么错？Network 只告诉你"后端崩了"，不告诉你为什么。
>
> 这种时候你可以直接把报错截图扔给 AI。AI 大概率能帮你修好——但你得等它回复、看它的解释、判断它说得对不对。来回可能要 1-2 分钟。
>
> 我教你一个更快的方式：**打一行日志，10 秒钟自己定位。**

**打开 `server/index.ts`，在 PUT handler 里加一行日志**：

```typescript
app.put("/api/issues/:id", (req, res) => {
  console.log("req.body:", req.body);       // ← 就加这一行

  const id = Number(req.params.id);
  const { status } = req.body;
  // ...
```

**保存（tsx watch 自动重启），再点一次按钮。**

**切到后端终端**：

```
req.body: undefined
```

**话术**：

> 一行日志，一眼看出来：`req.body` 是 `undefined`。
>
> 前端明明发了 `{ status: "已完成" }`，但后端收到的是 `undefined`——数据到了后端，但后端**不知道怎么解析**。
>
> 这是 Express 的一个常见问题。Express 默认不解析 JSON——你需要告诉它"如果前端发 JSON，帮我自动解析"。靠的就是这一行：

**指向第 8 行**：

```typescript
// app.use(express.json());  // ← 被注释掉了
```

> 取消注释：

```typescript
app.use(express.json());
```

**保存，再点"已完成"——状态变了！**

**话术**：

> 整个过程多长时间？加一行 `console.log`，看一眼终端，发现 `req.body` 是 `undefined`，联想到 `express.json()` 没开——**10 秒钟**。
>
> 你当然也可以把报错扔给 AI，AI 大概率也能告诉你答案。但关键区别是：
>
> - **扔给 AI**：等回复 → 读解释 → 判断对不对 → 可能还要追问。1-2 分钟。
> - **自己打日志**：加一行 `console.log` → 看输出 → 定位问题。10 秒。
>
> 而且你自己定位了一次之后，下次遇到同样的问题你就记住了。**扔给 AI 你学不到东西，自己看一遍你就记住了。**
>
> 当然，如果打了日志还是看不懂——那就把**日志的输出**发给 AI。比直接发"按钮点了没反应"有用得多，因为你已经帮 AI 缩小了范围。

### 0:30-0:33 — 补充：服务端报错怎么看

**目的**：快速补充一个场景——后端代码本身报错（不是 req.body 的问题，而是代码写错导致 crash）。让学员知道"终端不只是看日志，也能看报错"。

**话术**：

> 刚才两个 bug 都是"代码能跑，但行为不对"。还有一种情况更直接——**后端代码本身就跑不起来，直接崩了。**
>
> 比如你在后端写了这样一行：

**在 `server/index.ts` 任意位置加一行故意写错的代码**（现场演示，不改源文件）：

```typescript
issues.forEach(item => item.name.toUpperCase());  // item 上没有 name 属性
```

**保存，看终端**：

```
TypeError: Cannot read properties of undefined (reading 'toUpperCase')
    at server/index.ts:20:35
```

> 看到了吗？终端直接打出了报错——`TypeError`，在 `server/index.ts` 第 20 行。
>
> 跟浏览器 Console 里的报错结构一模一样：**类型 + 描述 + 文件行号**。读法也一样：看第一行，找行号。
>
> 所以前端报错看 Console，后端报错看终端——**读法是一样的**。

**撤销刚才加的那行代码，恢复正常。**

> 记住：后端没反应的时候，**第一件事去看终端**。如果终端有红色报错，读法跟 Console 一样。

### 0:37-0:42 — 部署概念：从 localhost 到公网 URL

**目的**：用最简单的话解释"部署是什么"，为接下来的实操做铺垫。

**话术**：

> 调试讲完了。接下来讲第二件事——**部署**。
>
> 我们的 Issue 看板现在跑在 `localhost:5173`。`localhost` 就是"你自己的电脑"——只有你能打开这个地址，别人打开看到的是空白。
>
> **部署就是把你的代码放到别人的电脑上跑。** 这个"别人的电脑"是云服务器——一台 24 小时开机、接着互联网的电脑。代码放上去之后，它会给你一个公网 URL，任何人打开浏览器访问这个 URL，就能用你的应用。
>
> 类比：你在家做了一道菜（localhost），只有你自己能吃。你把菜送到一个餐厅（云服务器），餐厅给你一个地址，朋友去餐厅就能吃到了。
>
> 部署有很多种方式——买服务器自己配、用 Docker、用各种云平台。我们今天用 **Vercel**，原因很简单：**它是最傻瓜的方式**——连上 GitHub 仓库，它自动帮你构建、部署、给 URL。不需要你配置服务器。

**关于后端和数据库**：

> 我们的项目有前端也有后端。部署的时候有三个问题要解决：
>
> 1. **前端怎么上线？** Vercel 帮你打包，放到它的服务器上。
> 2. **后端怎么上线？** 项目里有一个 `api/` 文件夹，Vercel 自动把里面的函数变成后端接口。
> 3. **数据存在哪？** 在本地，数据存在 Express 的变量里，重启就没了。线上也一样——如果不用数据库，数据随时会丢。
>
> 所以我们要加一个数据库。听起来很复杂，但 Vercel 提供了一个叫 **KV** 的服务——你可以理解为一个云上的"储物柜"，把数据存进去，下次来取还在。
>
> 我来演示一遍完整流程：从推代码、配数据库、到部署上线。

### 0:42-0:55 — 部署实操：Vercel + 数据库

**目的**：讲师全程直播完整部署流程，包含数据库配置。学员看到从本地到"任何人都能用、数据不会丢"的完整闭环。

**话术**：

> 现在来实际操作。

#### 第一步：推代码到 GitHub（1 分钟）

**讲师操作**：

```bash
cd lessons/lesson-5
git add .
git commit -m "第 5 课项目代码"
git push
```

**话术**：

> 第一步——把代码推到 GitHub。这个你们第 2 课学过，三步走。

#### 第二步：在 Vercel 上创建项目（2 分钟）

**讲师操作**：

1. 打开浏览器，访问 `vercel.com`，登录（用 GitHub 账号）
2. 点击 "Add New..." → "Project"
3. 选择课程仓库
4. **Root Directory** 填 `lessons/lesson-5`
5. Framework Preset：Vite（自动检测）
6. 点 "Deploy"

**话术**：

> Vercel 网站，GitHub 登录，选仓库，Root Directory 设成 `lessons/lesson-5`，Deploy。
>
> 等它构建完——

#### 第三步：加数据库（3 分钟）

**课前已准备好**：在 Upstash（upstash.com）上创建了一个免费 Redis 数据库，拿到了 `UPSTASH_REDIS_REST_URL` 和 `UPSTASH_REDIS_REST_TOKEN`。

**在 Vercel 项目控制台操作**：

1. 点左侧 "Settings" → "Environment Variables"
2. 添加两个变量：`UPSTASH_REDIS_REST_URL` 和 `UPSTASH_REDIS_REST_TOKEN`
3. 三个环境（Production / Preview / Development）都勾上

**话术**：

> 部署成功了，但数据还没有地方存。我们加一个数据库。
>
> 我提前在 Upstash 上创建了一个免费的 Redis 数据库。Redis 是什么？你可以理解为一个云上的"储物柜"——你用一个名字存数据进去，下次用同一个名字就能取出来。
>
> 现在把数据库的连接信息告诉 Vercel——在 Settings 里加两个环境变量，一个是地址，一个是密码。

#### 第四步：重新部署让数据库生效（2 分钟）

**讲师操作**：

1. 在 Vercel 项目页面，点 "Deployments" 标签
2. 找到最近一次部署，点右侧 "..." → "Redeploy"
3. 等待重新构建

**话术**：

> 环境变量加好了，重新部署一次，让代码读到数据库的配置。
>
> 点 Redeploy——等 1 分钟。

#### 第五步：验证——数据真的存住了（3 分钟）

**讲师操作**：

1. 打开部署好的 URL
2. Issue 看板正常显示
3. 点"已完成"改一张卡片的状态
4. **刷新页面**——状态还在！
5. **打开手机浏览器**（或让一个学员打开同一个 URL）——数据一致！

**话术**：

> 来，打开线上的 Issue 看板。
>
> 卡片出来了。我点"已完成"——状态变了。
>
> 现在刷新页面——
>
> **状态还在！** 不像本地版那样刷新就丢了。因为数据存在数据库里了。
>
> 我把这个 URL 发到群里——你们在自己手机上打开看看，是不是也能看到我刚改的状态？

**（等学员在手机上打开确认）**

> 看到了吧？你在电脑上改的状态，别人在手机上也能看到。这就是数据库的作用——**所有人访问的是同一份数据**。
>
> 回忆一下第 4 课开头讲的 Meego 类比——你改了一个 Bug 的状态，同事刷新也能看到。现在你们自己做的 Issue 看板也是这样了。

#### 第六步：对比完整链路（1 分钟）

**话术**：

> 整个链路走完了。从第 3 课到现在：
>
> - **第 3 课**：数据写死在前端 → 刷新就丢
> - **第 4 课**：数据搬到后端 → 刷新不丢了，但只有你能用
> - **今天**：数据存到数据库 + 部署到 Vercel → 刷新不丢，谁都能用
>
> **这就是一个真实的 Web 应用。** 前端显示页面，后端处理逻辑，数据库存数据，部署到云上让所有人访问。

**话术**：

**板书 / 投屏**：

```
第 1 课  全局地图      代码跑在哪？（浏览器、服务器、DevTools）
```

## 关键原则提醒

1. **调试是主角，部署是配角。** 如果时间不够，压缩部署而不是压缩调试。调试是学员课后每天都需要的技能。
2. **项目自带 bug，不需要现场制造。** 开场直接跑项目发现白屏，比"假设遇到报错"更有冲击力，也更可控。
3. **教方法，不教记忆。** 不要求学员记住每种错误码的含义。教的是一个可重复的流程：看 Console → 看 Network → 打日志 → 喂给 AI。
4. **"打日志 → 喂给 AI"是核心工作流。** 这是学员课后真正会用的调试方式——先自己缩小问题范围，再让 AI 精准回答。比直接把整段报错扔给 AI 高效得多。
5. **提问 cue 学员主动思考。** 每一步都用提问让学员口头回答（"Console 说了什么？""状态码多少？"），保持注意力但不引入操作风险。
6. **部署不解释 Serverless 原理。** `api/issues.ts` 一笔带过——"已经写好了，Vercel 自动处理"。学员只需要知道部署的流程，不需要理解 Serverless 的工作机制。
7. **前端代码零改动是最好的教学点。** 从本地到部署，`src/` 里的代码一个字都不用改。这是最直观的"好架构"体现。
8. **呼应第一课。** 第一课教了 DevTools，今天用 DevTools 来调试；第一课讲了浏览器到服务器的链路，今天把项目部署到了服务器上。形成闭环。
9. **这是最后一课，收束感很重要。** 不是"下课了"，而是"你们已经有了完整的能力链路"。结尾给方向感，让学员知道接下来可以自己继续。
