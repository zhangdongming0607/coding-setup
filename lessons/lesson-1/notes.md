# 第 1 课笔记：代码到底在哪里运行

## 一、一个网页是怎么出来的

你在浏览器输入网址，背后发生了这些事：

1. 浏览器跑去服务器要文件（HTML、CSS、JS）
2. 拿到文件后，浏览器把页面画出来
3. 页面画完了，代码还会偷偷再去服务器拿数据（比如用户信息、聊天记录）

一句话总结：**浏览器 → 服务器 → 浏览器**，来回倒。

## 二、前端、后端、配置文件

| 类型 | 跑在哪 | 干什么 | 长什么样 |
|---|---|---|---|
| 前端 | 浏览器里 | 画页面、处理点击 | `.tsx` `.css` `.html` |
| 后端 | 你的电脑 / 服务器上 | 存数据、算逻辑 | `server/index.ts` |
| 配置 | 哪都不跑，给工具看的 | 告诉工具怎么启动、怎么打包 | `package.json` `vite.config.ts` |

前端和后端靠 **API 请求**互相说话——前端发请求，后端回数据。

**怎么判断页面上的内容是前端给的还是后端给的？** 打开 Network 面板，刷新页面，看 API 请求的 Response——出现在 Response 里的数据就是后端返回的，其余的（比如按钮文字、页面标题、布局）都是前端代码里写好的。

一般来说：固定不变的内容（文案、布局、样式）放前端，会变的数据（用户信息、列表内容、聊天记录）放后端。

## 三、DevTools 三板斧

打开方式：`Cmd+Option+I`（Mac）/ `F12`（Windows）

| 面板 | 能看到什么 | 什么时候打开 |
|---|---|---|
| **元素 Elements** | 页面的 HTML 骨架和样式，可以现场改着试 | 想调样式的时候 |
| **网络 Network** | 页面发了哪些请求、成功没、返回了啥 | 搞不清是前端的锅还是后端的锅 |
| **控制台 Console** | 报错信息，红色的就是出事了；也可以自己加日志看数据 | 遇到问题，不确定是哪部分出了问题 |

想在代码里加日志，跟 AI 说：

```
帮我在 xxx 函数里加一行 console.log，把 xxx 变量打印出来
```

## 四、状态码怎么看

- **2 开头** — 没问题
- **4 开头** — 你这边有问题（404 地址不对，400 参数不对）
- **5 开头** — 服务器那边炸了

## 五、Git 操作

跟 AI 说这些话就行：

```
1. 帮我拉取最新代码
2. 帮我创建一个新分支，分支名叫 homework/我的名字/lesson-1
3. 我已经把作业写好放到 homework/我的目录/ 里了，帮我提交，说明是"第 1 课作业"
4. 帮我推到 GitHub 上
```

## 六、提 Pull Request（PR）

为什么不直接 push 到 main？两个原因：

1. **像交作业给老师批改** — 你先把作业放到一边，老师看过没问题了再合进去。工作中也是这样，代码写完先提 PR 让别人帮你看一眼。
2. **避免冲突** — 大家都往 main 直接 push，很容易互相覆盖。各自在自己的分支上写，最后通过 PR 合并，就不会打架。

这一步要自己在网页上操作：

1. 打开 https://github.com/zhangdongming0607/coding-setup
2. 页面顶部会出现黄色提示条 **"Compare & pull request"**，点它
3. 确认 base 是 `main`，compare 是你的分支
4. 标题写 `第 1 课作业 - 你的名字`
5. 点 **Create pull request**

注意：提完 PR 就行了，不要点那个绿色的 **Merge pull request** 按钮，等老师review。

## 七、Vibe Coding 小贴士

**多提交 = 多存档。**

用 AI 写代码的时候，每做完一小步就让 AI 帮你提交一次，相当于游戏存档。改崩了随时回档，不用从头再来。

```
帮我提交一下当前的改动
```
