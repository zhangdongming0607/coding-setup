# 第 4 课：后端 — 数据存在哪、API 是什么

> 目标：让学员理解"前端和后端的分工"、"API 是前后端之间的约定"，并亲手把第 3 课的 Issue 看板从"假数据"升级为"从后端拿数据"。

## 设计思路

第 3 课学员做了一个 Issue 看板，数据写死在 `useState` 里。这节课的核心动作是：**把数据从前端搬到后端**。

这个过程自然引出三个问题：
1. 数据为什么不能一直写在前端？（动机）
2. 后端怎么把数据给前端？（API）
3. 前端怎么拿到后端的数据？（fetch）

整节课围绕一个故事线：**Issue 看板的数据搬家**。数据只存在 JavaScript 变量里，不讲数据库、不讲 ORM、不讲 Redis。

### 0:00-0:07 — 开场：数据写死的问题

**目的**：从第 3 课的 Issue 看板切入，让学员自己发现"数据写死"的问题，并连接到他们使用 AI 的真实体验。

**动作**：

1. 打开浏览器，访问第 3 课的 Issue 看板（`localhost:5173`）
2. 点几下状态按钮，把"登录功能异常"改成"已完成"
3. 刷新页面

**话术**：

> 上节课我们做了这个 Issue 看板。大家看，我现在把"登录功能异常"改成"已完成"了。
>
> 现在我刷新一下页面——
>
> （刷新）
>
> 没了。又变回"待处理"了。
>
> 为什么？我们打开代码看看。

**打开 `App.tsx`，指向第 8-14 行**：

```typescript
const [issues, setIssues] = useState([
  {label: 'Bug', title: '首页加载速度太慢', assignee: '小明', status: '进行中'},
  {label: 'Bug', title: '登录功能异常', assignee: '小明', status: '待处理'},
  // ...
])
```

> 数据写死在代码里了。每次刷新页面，React 重新执行这个函数，`useState` 的初始值又是这个固定的数组。你改的那些状态，只存在页面的"临时记忆"（state）里，刷新就没了。

**先建立直觉——为什么要有后端**：

> 想象你们用的 Meego。你在上面改了一个 Bug 的状态，团队里所有人刷新页面都能看到——不是因为你的电脑跟每个人的电脑直接连着，而是改动先发到 Meego 的服务器，服务器再返给所有人。如果没有服务器，你只能把数据用 U 盘拷给别人。
>
> 这就是**后端**存在的意义：互联网上有无数个互不认识的客户端，需要一个"所有人都连着的中间人"来存数据、同步状态、定规则。
>
> 今天我们做的 Issue 看板，目标就是一样——你改了一个 Bug 的状态，你的同事刷新页面也能看到，靠的就是这个中间人。

**连接 AI 使用场景**：

> 你们平时让 AI 生成项目的时候，有没有遇到过这种情况：AI 帮你写了后端代码，但跑起来报错了，AI 跟你说"改一下 `routes/api.js` 第 15 行"，你打开一看完全看不懂——不知道那个文件在干嘛，也不知道怎么改。
>
> 今天这节课就是让你看得懂那些代码。我们要把数据从前端搬到一个**独立的地方**——**后端**。搬完之后，你就知道 AI 生成的后端代码到底在干嘛了。

### 0:07-0:14 — 概念：前端和后端各管什么

**目的**：用 Meego 类比建立"前后端分工"的心智模型。

**话术**：

> 在讲怎么做之前，先搞清楚一个概念：前端和后端到底怎么分工。
>
> 大家都用过 Meego——Meego 就是最典型的前后端分离应用。
>
> **前端 = Meego 客户端**
> - Bug 列表长什么样？状态按钮在哪？筛选框怎么显示？
> - 你怎么改状态？怎么新建 Issue？
> - 这些都是前端管的事——**你看到的一切、你的操作，都是前端**
>
> **后端 = Meego 服务器**
> - 所有 Bug 数据存在哪？谁负责哪个模块？历史记录怎么查？
> - 你改了一个状态，怎么让别人也看到？
> - 这些是后端管的事——**数据存在哪、业务逻辑怎么算，都是后端**
>
> **API = 客户端和服务器之间的通信协议**
> - 你点"已完成"，客户端按什么格式打包、发到哪个地址、服务器怎么回
> - **API 就是这个约定——前端和后端之间的通信规则**

**板书**（白板或飞书文档）：

```
┌─────────────────┐                        ┌─────────────────┐
│ 前端（Meego客户端）│     API（通信协议）      │ 后端（Meego服务器）│
│                 │                        │                 │
│  显示页面        │ ──── 请求 ──────→      │  存数据          │
│  响应用户操作    │                        │  处理业务逻辑     │
│  更新界面        │ ←─── 响应（数据）──     │  返回结果         │
└─────────────────┘                        └─────────────────┘
```

> 你在 Meego 客户端点"已完成"，客户端（前端）通过 API 把改动发给 Meego 服务器（后端），服务器存好之后，你同事的客户端刷新一下也能看到。当时我们在 Network 面板里看到的那些请求，就是 API 调用。
>
> 今天要做的事情一样：让 Issue 看板的前端通过 API 去后端拿数据，而不是自己写死。

### 0:14-0:18 — 开两个终端：模拟 Meego 客户端和服务器

**目的**：让学员直观感受"前端和后端是两个独立运行的程序"，并教会同时启动两个服务。

**话术**：

> 在开始之前，先说两件事。
>
> **第一，这两个命令你不需要记，让 AI 帮你跑就行。** 你告诉 AI "帮我把项目跑起来"，它会告诉你敲什么命令。今天你只需要看懂它在做什么——前端起一个服务，后端起一个服务。
>
> **第二，为什么前端也要"跑服务"？** 你们平时双击一个 HTML 文件，浏览器直接打开，不需要跑任何东西。但我们现在用的是 React——React 的代码是 TypeScript，浏览器看不懂，需要先"翻译"成普通 JavaScript。`npm run dev` 就是启动这个翻译程序，同时帮你开一个本地服务器来托管页面。所以你看到的 `localhost:5173`，本质上就是一个跑在你电脑上的小型网站服务器。
>
> 好，现在来起服务。前端和后端是**两个独立的程序**，要分别启动。就像 Meego 客户端和 Meego 服务器是两个独立的东西——两边都得跑，数据才能同步。
>
> 真实场景里，Meego 的服务器是一直跑着的，部署在云上，你不需要管它。你打开 Meego 客户端，背后的服务器早就在那等着了。
>
> 今天我们是在本地模拟这件事——所以需要自己手动把服务器跑起来。我们要开两个终端窗口：一个模拟 **Meego 客户端**（前端），一个模拟 **Meego 服务器**（后端）。

**演示步骤**：

1. 在 VS Code 中打开终端（`` Ctrl+` ``）
2. 点终端右上角的 `+` 号，新开一个终端标签页
3. 指着两个标签页说：

> 左边这个是 **Meego 客户端**，跑前端：
> ```
> cd lessons/lesson-4
> npm run dev
> ```
> 看到 `localhost:5173` 就说明 Meego 客户端跑起来了。
>
> 右边这个是 **Meego 服务器**，跑后端：
> ```
> cd lessons/lesson-4/server
> npx tsx watch index.ts
> ```
> 看到"后端跑在 http://localhost:3001"就说明服务器开工了。
>
> 两个端口号不一样——5173 是客户端的门牌号，3001 是服务器的门牌号。前端跑在 5173，后端跑在 3001，它们通过 API 互相通信。
>
> 以后你们自己做项目，遇到要同时启动前后端的情况，就开两个终端。

### 0:18-0:22 — API 到底长什么样

**目的**：让学员直观看到 API 的输入和输出，理解"路径 + 方法 + JSON"三要素。

**话术**：

> API 听起来很抽象，但我现在让你们看一个活的。

**动作**：

打开浏览器，在地址栏直接输入 `http://localhost:3001/api/issues`，回车。

**话术**：

> 看到了吗？浏览器显示了一段数据。这就是后端返回的 Issue 列表。
>
> 你刚才做的事情，就是手动调了一次 API。跟你平时在浏览器里输入 baidu.com 没有本质区别——都是浏览器去找一个服务器要东西。区别只是 baidu.com 返回的是一个网页，而这个地址返回的是**纯数据**。

**指着浏览器里的 JSON 数据**：

```json
[
  { "id": 1, "label": "Bug", "title": "首页加载速度太慢", "assignee": "小明", "status": "进行中" },
  { "id": 2, "label": "Bug", "title": "登录功能异常", "assignee": "小明", "status": "待处理" }
]
```

> 这个数据格式叫 **JSON**。你们其实已经见过了——它长得跟第 3 课 `useState` 里写的那些数据一模一样。方括号 `[]` 是数组（一组东西），花括号 `{}` 是对象（一个东西的详细信息）。
>
> 每个 API 有三个要素：
>
> | API 要素 | Meego 里的例子 | 代码里的例子 |
> |---------|---------|------|
> | **路径（URL）** | 你在操作哪个资源（Issue 列表） | `/api/issues` |
> | **方法（Method）** | 你要做什么操作 | `GET`（查列表）、`POST`（新建）、`PUT`（改状态）、`DELETE`（删除）|
> | **数据（JSON）** | 改成什么状态（比如"已完成"） | `{ "status": "已完成" }` |
>
> 刚才你在浏览器地址栏输入 URL 回车，浏览器默认用的就是 **GET** 方法——"给我看看数据"。

**重要衔接**：

> 等一下我们会看到，前端代码里用 `fetch` 函数做的事情，跟你刚才在地址栏输入 URL 做的事情，本质上是一样的——都是去后端请求数据。只不过 `fetch` 是代码自动做的，不需要你手动输入地址。

### 0:22-0:35 — 看代码：后端长什么样

**目的**：打开后端代码，逐块讲解。每讲一块就 cue 学员互动，保持注意力。

**话术**：

> 刚才我们在浏览器里看到了 API 返回的数据。现在我们去看看后端代码——那些数据是哪里来的？

**打开 `server/index.ts`，逐块讲解**：

#### 第一块：开店准备（第 1-5 行）

```typescript
import express from "express";

const app = express();
app.use(express.json());
```

> 两件事：
> 1. 引入 Express。先说 Node.js——JavaScript 本来只能在浏览器里跑，Node.js 把它解放出来，让它能跑在服务器上。Express 是 Node.js 上最流行的后端框架，帮你管所有的 API，相当于一个现成的脚手架。
> 2. 创建一个 `app`，然后告诉它"如果前端发来 JSON 数据，帮我自动解析"。
>
> 顺便说一下——后端用什么语言写其实很自由。Node.js 只是其中一种，Go、Java、Python 都可以写后端，你听说过的各种后端技术栈都是这回事。比如我们公司非联的后端大部分就是用 Go 写的。今天用 Node.js 是因为你们已经会 JavaScript 了，换个语言逻辑是一模一样的。
>
> 这几行是固定写法，每个 Express 后端都长这样，不需要记。

#### 第二块：数据（第 7-15 行）

```typescript
const issues = [
  { id: 1, label: "Bug", title: "首页加载速度太慢", assignee: "小明", status: "进行中" },
  { id: 2, label: "Bug", title: "登录功能异常", assignee: "小明", status: "待处理" },
  { id: 3, label: "Bug", title: "注册功能异常", assignee: "小明", status: "进行中" },
  { id: 4, label: "Bug", title: "忘记密码功能异常", assignee: "小明", status: "待处理" },
  { id: 5, label: "Bug", title: "用户中心功能异常", assignee: "小明", status: "进行中" },
];
```

> 眼熟吗？

**（cue 学员）**：「这段数据你们之前在哪个文件里见过？」

等学员回答（App.tsx / useState 里）。

> 对，**这就是第 3 课 App.tsx 里 `useState` 那段数据**。一模一样，只是搬到了后端。
>
> 多了一个 `id` 字段——每条 Issue 有一个唯一编号。为什么需要 id？上节课我们用 index（第几条）来标识，但 index 有个问题：如果你删了第 2 条，后面每条的 index 都变了。id 是每条数据自己的**身份证号**，不管你怎么删除、排序，它都不会变。
>
> 注意：这个数据现在存在一个 JavaScript 变量里，跑在服务器的内存中。只要服务器不重启，数据就一直在。比前端好在哪？**刷新页面不会丢**——因为刷新页面只是重新加载前端，后端那个变量没有受影响。

（预判学员问题：「那服务器重启呢？」）

> 好问题——服务器重启确实也会丢。真实项目里数据会存在**数据库**里。数据库你可以理解为服务器的硬盘——内存里的东西断电就没了，但硬盘里的东西断电还在。数据库就是专门给服务器存数据用的"硬盘"，不管重启多少次，数据都还在。但今天我们不讲数据库，先用变量把流程跑通，理解 API 怎么工作是第一步。

#### 第三块：第一个 API —— 获取所有 Issue（第 17-20 行）

```typescript
// GET /api/issues —— 获取所有 Issue
app.get("/api/issues", (req, res) => {
  res.json(issues);
});
```

> 这就是你刚才在浏览器里访问的那个 API。拆开看：

**在白板上画**：

```
app .get ( "/api/issues" ,  (req, res) => { ... } )
 │    │         │                │
 │    │         │                └── 收到请求后做什么（函数）
 │    │         └── 路径：就是你在浏览器地址栏输入的那个
 │    └── 方法：GET = 拿数据
 └── Express 创建的服务器实例
```

> 翻译成人话：**当有人用 GET 方法访问 /api/issues 这个路径时，把 issues 数组以 JSON 格式返回。**
>
> `req` 是请求（request）——前端发过来的东西。`res` 是响应（response）——你要回给前端的东西。
>
> `res.json(issues)` 就是"把 issues 这个数组转成 JSON，返回给前端"。
>
> 这三行代码，就是一个完整的 API。是不是比想象的简单？

**（cue 学员验证）**：「大家回忆一下，你们刚才在浏览器里访问 `localhost:3001/api/issues` 看到了什么？那段 JSON 数据就是这三行代码返回的。能对上吧？」

#### 第四块：第二个 API —— 修改 Issue 状态（第 22-32 行）

> 在看这段代码之前，先发给学员一张速查卡——有几个 JS 语法在这里会出现，先看懂再看代码：

**发给学员（可以贴群里）**：

```
JS 语法速查——看懂这段代码
════════════════════════════════

① 对象是什么
   对象就是一个有层级的信息包。比如字节跳动的组织架构：
     字节跳动 → 火山引擎 → 基础设施部门 → 负责人：张三
   代码里的 req.params.id 就是：
     req（这次请求）→ params（路径参数）→ id（具体的值）
   用 . 一层层往里取，就像顺着公司架构找到具体的人。

② 数组是什么
   数组就是一个有序列表。Meego 里的 Bug 列表就是数组——
   一条 Bug、两条 Bug、三条 Bug，按顺序排着。
   issues.find(...) 就是在列表里逐条找，找到满足条件的那条返回。

③ 解构赋值
   const { status } = req.body
   等同于 const status = req.body.status，只是简写。

看不懂的代码？选中它，Cursor 里按 Cmd+L 直接问。
```

```typescript
// PUT /api/issues/:id —— 修改一个 Issue 的状态
app.put("/api/issues/:id", (req, res) => {
  const id = Number(req.params.id);
  const { status } = req.body;

  const issue = issues.find(item => item.id === id);
  if (!issue) {
    res.status(404).json({ error: "Issue 不存在" });
    return;
  }

  issue.status = status;
  res.json(issue);
});
```

> 这个 API 复杂一点，但逻辑很清楚。

**（cue 学员）**：「`app.put` 的 put 是什么意思？我们刚才讲过 API 三要素里的"方法"——GET 是拿数据，那 PUT 呢？」

等学员回答（改数据）。

> 对，PUT 是**改数据**。我一行一行翻译：
>
> 1. `"/api/issues/:id"` —— 路径里的 `:id` 是个占位符。如果前端请求 `/api/issues/3`，那 `req.params.id` 就是 `3`。就像你跟服务员说"3 号桌"，服务员就知道是哪桌
> 2. `req.body` —— 前端发过来的数据。这里前端会告诉后端"要改成什么状态"
> 3. `issues.find(...)` —— 在数组里找到 id 对应的那条 Issue
> 4. 找不到就返回 404（"找不到"）——还记得第 1 课讲的吗？4 开头是"你的问题"
> 5. 找到了就改 status，然后把改完的结果返回

**（cue 学员）**：「用餐厅类比的话，这个 API 在做什么？」

期望回答方向：客人跟服务员说"3 号桌的菜改成微辣"，服务员找到 3 号桌的单子，改好，回来说"好了"。

#### 第五块：开门营业（最后两行）

```typescript
app.listen(3001, () => {
  console.log("后端跑在 http://localhost:3001");
});
```

> 最后这一步：让服务器开始监听 3001 端口。就是刚才我们在终端里看到的那行输出。3001 就是**厨房的门牌号**，前端通过这个门牌号来找后端。

**总结后端代码**：

**（cue 学员）**：「整个后端代码做了几件事？谁来总结一下？」

期望回答方向：
1. 开店（创建 Express app）
2. 定菜单（写 API：GET 拿数据、PUT 改数据）
3. 开门（监听端口）

> 没错。后端代码比前端简单——它不用管页面长什么样，只管"你要什么数据，我给你什么数据"。

### 0:35-0:45 — Live Coding：前端对接后端

**目的**：现场修改第 3 课的前端代码，让它从后端拿数据。这是全课的核心环节。

**话术**：

> 后端准备好了，API 也测过了。现在我们来改前端——让 Issue 看板不再用假数据，而是从后端拿真数据。

#### 改动 1：页面加载时从后端获取数据（0:35-0:39）

**话术**：

> 第一步：把写死的数据删掉，换成从后端拿。

**打开 `App.tsx`，指向当前代码**：

```typescript
// 现在的代码——数据写死了
const [issues, setIssues] = useState([
  {label: 'Bug', title: '首页加载速度太慢', assignee: '小明', status: '进行中'},
  {label: 'Bug', title: '登录功能异常', assignee: '小明', status: '待处理'},
  // ...
])
```

**改成**：

```typescript
import { useState, useEffect } from 'react';

// 改成空数组——一开始没有数据
const [issues, setIssues] = useState([]);

// 页面加载后，去后端拿数据
useEffect(() => {
  fetch("/api/issues")
    .then(res => res.json())
    .then(data => setIssues(data));
}, []);
```

> 改了两件事：
>
> **第一**，`useState` 的初始值从一大堆数据变成了空数组 `[]`。页面刚加载的时候，没有 Issue——因为数据还没从后端拿到。
>
> **第二**，加了 `useEffect`。还记得第 3 课讲的吗？`useEffect` 是"页面加载后做一件事"。这里做的事情是：
>
> ```
> fetch("/api/issues")          ← 去后端的 /api/issues 拿数据（就是刚才在浏览器里访问的那个地址）
>   .then(res => res.json())    ← 把返回的数据从 JSON 格式解析出来
>   .then(data => setIssues(data))  ← 把数据放到 state 里
> ```
>
> 三行代码，三步操作。**fetch 就是前端调 API 的方式**——之前你在浏览器地址栏手动输入 URL，现在代码帮你自动做了。
>
> 第三行 `setIssues(data)` 执行后，state 就变了。上节课讲的——**state 一变，页面自动更新**。所以数据一到，卡片就出现了。

**顺便说一句 Vite 代理**：

> 你们可能注意到了：代码里写的是 `fetch("/api/issues")`，不是 `fetch("http://localhost:3001/api/issues")`。为什么不用写完整地址？
>
> 因为我们在项目配置里设了一个**转发规则**——前端看到 `/api` 开头的请求，会自动帮你转发到后端的 3001 端口。就像餐厅大厅有个内部通道直通厨房，你不需要出门绕到后门去找厨房。
>
> 知道有这个东西就行，不需要深究。

**演示效果**：

保存文件，切到浏览器，页面应该自动刷新（Vite HMR）。

> 看，卡片出来了！但这些数据不是代码里写死的，而是从后端拿的。
>
> 怎么证明？打开 Network 面板。

**打开 Chrome DevTools → Network 面板 → 刷新页面**。

> 看到这个请求了吗？`/api/issues`，GET 方法，状态 200（成功）。点开看 Response——就是我们之前在浏览器里看到的那段 JSON 数据。
>
> 代码里的 `fetch` → Network 面板里的请求 → 后端的 `app.get` → 返回数据 → 前端 `setIssues` → 页面更新。整条链路通了。

#### 改动 2：修改状态时通知后端（0:39-0:45）

**话术**：

> 现在拿数据的问题解决了。但如果你点"已完成"按钮，然后刷新，状态还是会回去。为什么？

**（cue 学员）**：「谁来说说为什么？」

期望回答方向：因为点按钮只改了前端的 state，没有告诉后端。

> 对。我们要让前端点按钮时，先告诉后端"我要改状态"，后端改完了，前端再重新拿一遍最新数据。

**⚡ 讲师操作提示：这块不现场手写，用 Cursor 直接改**

> 在 Cursor 里选中当前的 `onChangeStatus` 整块，按 `Cmd+L`，发这条提示词：
>
> ```
> 把这个 onChangeStatus 改成：先用 fetch PUT /api/issues/${issue.id} 告诉后端新状态，
> 成功后再 fetch GET /api/issues 重新拿完整列表，然后 setIssues(data) 更新页面。
> 用 method PUT，headers Content-Type application/json，body JSON.stringify({ status: newStatus })
> ```
>
> AI 改完后你在屏幕上讲解这段代码就行，不需要现场手打。

**指向当前的 onChangeStatus**：

```typescript
// 现在的代码——只改了前端 state
onChangeStatus={(status) => {
  setIssues(() => {
    const newIssues = [...issues];
    newIssues[index] = {label: issue.label, title: issue.title, assignee: issue.assignee, status: status};
    return newIssues;
  })
}}
```

**改成**：

```typescript
// 先告诉后端，后端改完了再重新拿数据
onChangeStatus={(newStatus) => {
  fetch(`/api/issues/${issue.id}`, {
    method: "PUT",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ status: newStatus }),
  })
    .then(() => fetch("/api/issues"))
    .then(res => res.json())
    .then(data => setIssues(data));
}}
```

> 跟刚才的 fetch 比，这里多了几样东西：
>
> - **`method: "PUT"`** —— 不再是默认的 GET（拿数据），而是 PUT（改数据）
> - **`` `/api/issues/${issue.id}` ``** —— 路径里带了 Issue 的 id。反引号和 `${}` 是字符串拼接——如果 `issue.id` 是 3，最终路径就是 `/api/issues/3`
> - **`headers`** —— 告诉后端"我发给你的数据是 JSON 格式的"
> - **`body: JSON.stringify({ status: newStatus })`** —— 要改成什么状态
>
> 后端改完之后，前端做了什么？重新 `fetch("/api/issues")` **拿一遍最新的完整列表**，然后 `setIssues(data)` 更新页面。这个模式你们已经见过了——跟页面加载时拿数据是一模一样的。

**演示效果**：

1. 点击一张卡片的"已完成"按钮
2. 打开 Network 面板，指出新出现的请求

> 看 Network 面板——多了两个请求：
> 1. 一个 **PUT** 请求——前端告诉后端"改状态"
> 2. 一个 **GET** 请求——前端重新拿最新数据
>
> 现在关键的测试——**刷新页面**。
>
> （刷新）
>
> 状态还在！"已完成"没有消失。因为这次我们改的是后端的数据，刷新页面前端重新去后端拿数据，拿到的就是改过的版本。
>
> **对比一下课开头**：同样是刷新页面，开头数据丢了，现在数据还在。区别就在于——数据存在后端了。

| 卡点 | 应对 |
|------|------|

**答案公布**：

```typescript
app.post("/api/issues", (req, res) => {
  const { label, title, assignee } = req.body;        // ← 前端发来的数据

  const newIssue = {
    id: issues.length + 1,                              // ← 数组长度 + 1
    label,
    title,
    assignee,
    status: "待处理",                                    // ← 新 Issue 默认待处理
  };

  issues.push(newIssue);                                // ← 加到数组末尾
  res.json(newIssue);                                   // ← 返回 JSON
});
```

**答案讲解**：

> 跟 GET 的结构一模一样：`app.方法(路径, 处理函数)`。
>
> 区别在哪？
> - GET 是**给**数据，所以直接 `res.json(issues)` 返回整个数组
> - POST 是**收**数据，所以先从 `req.body` 里取前端发来的内容，组装成一条新 Issue，加到数组里，再返回
>
> 这三种 API（GET / POST / PUT）覆盖了"拿、增、改"三种操作。真实项目里还有 DELETE（删），你们作业里会用到。

### 0:53-0:56 — 总结

**话术**：

> 今天的核心就三件事：
>
> **1. 前端管显示，后端管数据。**
> 前端决定页面长什么样、用户怎么操作。后端决定数据存在哪、怎么改。它们通过 API 通信。
>
> **2. API = 路径 + 方法 + 数据。**
> - 路径：去哪找（`/api/issues`）
> - 方法：做什么（GET 拿、POST 加、PUT 改、DELETE 删）
> - 数据：用 JSON 格式传
>
> **3. 前端用 `fetch` 调 API。**
> ```
> fetch("/api/issues")                  ← 最简单的，GET 拿数据
> fetch("/api/issues", { method, body })  ← 带方法和数据，POST / PUT / DELETE
> ```

**速查卡**（发到群里）：

```
后端 & API 速查卡
═══════════════════════════

后端 = 管数据的程序（跑在服务器上）
API  = 前端和后端之间的约定（路径 + 方法 + 数据格式）
JSON = 前后端之间传数据的格式

常见 API 方法：
  GET    → 拿数据    → app.get("/路径", (req, res) => { ... })
  POST   → 加数据    → app.post("/路径", (req, res) => { ... })
  PUT    → 改数据    → app.put("/路径", (req, res) => { ... })
  DELETE → 删数据    → app.delete("/路径", (req, res) => { ... })

前端调 API：
  fetch("/api/xxx")                        → GET，拿数据
  fetch("/api/xxx", {                      → POST/PUT，发数据
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ ... })
  })

后端收数据：
  req.params.id  → 路径里的参数（/api/issues/:id 里的 id）
  req.body       → 前端发过来的 JSON 数据

后端发数据：
  res.json(数据)         → 返回 JSON
  res.status(404).json() → 返回错误状态码 + JSON
```

## 关键原则提醒

1. **一条故事线**：Issue 看板的数据搬家。所有内容围绕这件事展开，不讲数据库、不讲 ORM、不讲部署。
2. **延续第 3 课的项目**：学员对 Issue 看板已经熟悉，在已知的项目上做增量修改，认知负担最小。
3. **先看效果再看代码**：每个 API 先在浏览器里访问看到 JSON 数据，建立直觉后再看后端代码怎么写的。
4. **cue 学员而不是让学员动手**：后端代码讲解环节，每讲一块就 cue 学员回答问题，保持注意力但不引入操作风险。
5. **Network 面板是贯穿线索**：每次写完 API 或改完前端，都打开 Network 面板看一眼，强化"代码的效果是可观察的"。这也衔接了第 1 课教的 DevTools 用法。
6. **不讲底层**：不解释 HTTP 协议、不讲 Promise 原理、不讲 Express 中间件机制。fetch 三行代码当固定模板用，学员只需要知道"这样写就能拿到数据"。
7. **刷新是关键测试**：全课有两次"刷新页面"的对比——开头刷新数据丢了（引出问题），结尾刷新数据还在（证明解决了）。这个前后呼应是最有说服力的。
8. **动手环节用填空题**：不让学员从零写代码，给骨架让他们填关键部分，降低门槛。
