第 4 课：后端 — 数据存在哪、API 是什么

把 Issue 看板的假数据搬到后端，搞懂前后端分工和 API 通信。

4.1 问题：数据写死了，刷新就没了

上节课我们的 Issue 看板，数据写死在代码里：

const [issues, setIssues] = useState([
  {label: 'Bug', title: '首页加载速度太慢', assignee: '小明', status: '进行中'},
  {label: 'Bug', title: '登录功能异常', assignee: '小明', status: '待处理'},
  // ...
])

点按钮改了状态，刷新页面就没了——因为每次刷新，React 重新执行函数，useState 又从初始值开始。

想象你们用的 Meego。你在上面改了一个 Bug 的状态，团队里所有人刷新页面都能看到——不是因为你的电脑跟每个人的电脑直接连着，而是改动先发到 Meego 的服务器，服务器再返给所有人。如果没有服务器，你只能把数据用 U 盘拷给别人。

这就是后端存在的意义：互联网上有无数个互不认识的客户端，需要一个"所有人都连着的中间人"来存数据、同步状态、定规则。

今天我们做的 Issue 看板，目标就是一样——你改了一个 Bug 的状态，你的同事刷新页面也能看到，靠的就是这个中间人。

真实的应用里，数据不写在前端代码里。数据存在后端——一个独立的程序，专门管数据。前端要数据就问后端要，前端要改数据就告诉后端去改。

4.2 前端和后端各管什么

用 Meego 来类比：

角色 类比 职责
前端 Meego 客户端 Bug 列表长什么样、状态按钮在哪、筛选框怎么显示——你看到的一切、你的操作
后端 Meego 服务器 所有 Bug 数据存在哪、谁负责哪个模块、历史记录怎么查——数据存在哪、业务逻辑怎么算
API 客户端和服务器之间的通信协议 你点"已完成"，客户端按什么格式打包、发到哪个地址、服务器怎么回——前端和后端之间的通信规则 
┌─────────────────┐                        ┌─────────────────┐
│ 前端（Meego客户端）│      API（通信协议）     │ 后端（Meego服务器）│
│                 │                        │                 │
│  显示页面        │ ──── 请求 ──────→       │  存数据          │
│  响应用户操作     │                        │  处理业务逻辑     │
│  更新界面        │ ←─── 响应（数据）──      │  返回结果         │
└─────────────────┘                        └─────────────────┘

你在 Meego 客户端点"已完成"，客户端（前端）通过 API 把改动发给 Meego 服务器（后端），服务器存好之后，你同事的客户端刷新一下也能看到。你在 Network 面板里看到的那些请求，就是 API 调用。

4.3 两个终端：模拟 Meego 客户端和服务器

前端和后端是两个独立的程序，要分别启动。就像 Meego 客户端和 Meego 服务器是两个独立的东西——两边都得跑起来，数据才能同步。

真实场景里，Meego 的服务器部署在云上，一直在跑，你不需要管它。今天我们在本地模拟这件事，所以需要自己把两边都启动起来。

在 Cursor 里开两个终端标签页：

终端 角色 命令 端口
终端 1 Meego 客户端（前端） cd lessons/lesson-4 && npm run dev localhost:5173
终端 2 Meego 服务器（后端） cd lessons/lesson-4/server && npm run dev localhost:3001

两个端口号不一样——5173 是客户端的门牌号，3001 是服务器的门牌号。它们通过 API 互相通信。

这两个命令不需要记，让 AI 帮你跑就行。你只需要知道：前端起一个服务，后端起一个服务，两边都跑起来才能通信。

4.4 API 长什么样

API 不抽象——在浏览器地址栏输入 http://localhost:3001/api/issues，回车：

[
  { "id": 1, "label": "Bug", "title": "首页加载速度太慢", "assignee": "小明", "status": "进行中" },
  { "id": 2, "label": "Bug", "title": "登录功能异常", "assignee": "小明", "status": "待处理" },
  ...
]

你刚才做的事情，就是手动调了一次 API。跟输入 baidu.com 没有本质区别——都是浏览器去找服务器要东西。区别是 baidu.com 返回网页，这个地址返回纯数据。

API 三要素

要素 Meego 里的例子 代码里的例子
路径（URL） 你在操作哪个资源（Issue 列表） /api/issues
方法（Method） 你要做什么操作 GET（查列表）、POST（新建）、PUT（改状态）、DELETE（删除）
数据（JSON） 改成什么状态（比如"已完成"） { "status": "已完成" }

JSON 就是前后端之间传数据的格式——长得跟第 3 课 useState 里写的数据一模一样。方括号 [] 是数组（一组东西），花括号 {} 是对象（一个东西的详细信息）。

4.5 看代码：后端长什么样

打开 server/index.ts，整个文件做了三件事：启动服务、定义 API、监听端口。

第一块：启动服务

import express from "express";

const app = express();
app.use(express.json());

引入 Express。先说 Node.js——JavaScript 本来只能在浏览器里跑，Node.js 把它解放出来，让它能跑在服务器上。Express 是 Node.js 上最流行的后端框架，帮你管所有的 API。

顺便说一下——后端用什么语言写很自由。Node.js 只是其中一种，Go、Java、Python 都可以写后端。比如我们公司非联的后端大部分就是用 Go 写的。今天用 Node.js 是因为你们已经会 JavaScript 了，换个语言逻辑是一模一样的。

这几行是固定写法，不需要记。

第二块：数据

const issues = [
  { id: 1, label: "Bug", title: "首页加载速度太慢", assignee: "小明", status: "进行中" },
  { id: 2, label: "Bug", title: "登录功能异常", assignee: "小明", status: "待处理" },
  // ...
];

眼熟吗？就是第 3 课 App.tsx 里 useState 那段数据，搬到了后端。

多了一个 id 字段——每条 Issue 的身份证号。为什么不用 index？因为你删了第 2 条，后面每条的 index 都变了。id 不会因为删除或排序而改变。

数据存在 JavaScript 变量里，跑在服务器内存中。刷新页面不会丢（刷新只重新加载前端，后端的变量不受影响）。但服务器重启就没了——数据库你可以理解为服务器的硬盘，内存里的东西断电就没了，但硬盘里的数据断电还在。真实项目会用数据库，今天先用变量把流程跑通。

第三块：API — 获取所有 Issue

// GET /api/issues —— 获取所有 Issue
app.get("/api/issues", (req, res) => {
  res.json(issues);
});

这就是你刚才在浏览器里访问的那个 API。拆开看：

app .get ( "/api/issues" ,  (req, res) => { ... } )
 │    │         │                │
 │    │         │                └── 收到请求后做什么（函数）
 │    │         └── 路径（就是浏览器地址栏输入的那个）
 │    └── 方法：GET = 拿数据
 └── Express 创建的服务器实例

翻译成人话：当有人用 GET 方法访问 /api/issues 时，把 issues 数组以 JSON 格式返回。

  • req = request，前端发过来的东西
  • res = response，你要回给前端的东西
  • res.json(issues) = 把数组转成 JSON 返回

三行代码，一个完整的 API。

第四块：API — 修改 Issue 状态

在看这段代码之前，先认识几个 JS 语法：

📋 JS 语法速查——看懂这段代码

① 对象是什么
对象就是一个有层级的信息包。比如字节跳动的组织架构：
字节跳动 → 火山引擎 → 基础设施部门 → 负责人：张三
代码里的 req.params.id 就是：
req（这次请求）→ params（路径参数）→ id（具体的值）
用 . 一层层往里取，就像顺着公司架构找到具体的人。

② 数组是什么
数组就是一个有序列表。Meego 里的 Bug 列表就是数组——一条 Bug、两条 Bug、三条 Bug，按顺序排着。
issues.find(...) 就是在列表里逐条找，找到满足条件的那条返回。

③ 解构赋值
const { status } = req.body
等同于 const status = req.body.status，只是简写。

💡 看不懂的代码？选中它，Cursor 里按 Cmd+L 直接问。

// PUT /api/issues/:id —— 修改一个 Issue 的状态
app.put("/api/issues/:id", (req, res) => {
  const id = Number(req.params.id);
  const { status } = req.body;

  const issue = issues.find(item => item.id === id);
  if (!issue) {
    res.status(404).json({ error: "Issue 不存在" });
    return;
  }

  issue.status = status;
  res.json(issue);
});

逐行翻译：

  1. app.put — PUT 方法，表示"修改数据"
  2. "/api/issues/:id" — :id 是占位符。请求 /api/issues/3 时，req.params.id 就是 3
  3. req.body — 前端发过来的数据（"要改成什么状态"）
  4. issues.find(...) — 在数组里找到对应的那条 Issue
  5. 找不到就返回 404（"找不到"）——第 1 课讲的，4 开头是"你的问题"
  6. 找到了就改 status，把改完的结果返回

第五块：监听端口

app.listen(3001, () => {
  console.log("后端跑在 http://localhost:3001");
});

让服务器开始监听 3001 端口——3001 是服务器的门牌号，前端通过这个门牌号来找后端。

总结：整个后端代码就三件事——启动服务（创建 app）、定义 API（写 GET/PUT）、监听端口（开始接收请求）。后端比前端简单，它不管页面长什么样，只管"你要什么数据，我给你什么数据"。

4.6 前端对接后端

后端 API 准备好了，现在改前端——让 Issue 看板从后端拿数据。

改动 1：页面加载时从后端获取数据

// 之前：数据写死
const [issues, setIssues] = useState([
  {label: 'Bug', title: '首页加载速度太慢', ...},
  ...
])

// 之后：从后端获取
import { useState, useEffect } from 'react';

const [issues, setIssues] = useState([]);

useEffect(() => {
  fetch("/api/issues")
    .then(res => res.json())
    .then(data => setIssues(data));
}, []);

三行 fetch，三步操作：

代码 做了什么
fetch("/api/issues") 去后端拿数据（就是你在浏览器地址栏输入的那个地址）
.then(res => res.json()) 把返回的 JSON 解析出来
.then(data => setIssues(data)) 放到 state 里 → 页面自动更新

fetch 就是前端调 API 的方式——之前你在浏览器地址栏手动输入 URL，现在代码自动做了。

关于地址：代码里写 fetch("/api/issues")，不是 fetch("http://localhost:3001/api/issues")。因为项目配了一个转发规则（Vite 代理）——前端看到 /api 开头的请求，自动转发到后端 3001 端口。知道有这个东西就行，不需要深究。

打开 Network 面板验证：能看到 /api/issues 请求，GET 方法，状态 200。

改动 2：修改状态时通知后端

现在能拿数据了，但点"已完成"按钮后刷新，状态还是回去——因为只改了前端 state，没告诉后端。

// 之前：只改前端 state
onChangeStatus={(status) => {
  setIssues(() => {
    const newIssues = [...issues];
    newIssues[index] = { ...issue, status };
    return newIssues;
  })
}}

// 之后：先告诉后端，再重新拿数据
onChangeStatus={(newStatus) => {
  fetch(`/api/issues/${issue.id}`, {
    method: "PUT",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ status: newStatus }),
  })
    .then(() => fetch("/api/issues"))
    .then(res => res.json())
    .then(data => setIssues(data));
}}

跟刚才的 fetch 比，多了几样东西：

  • method: "PUT" — 不再是默认的 GET，而是 PUT（改数据）
  • `/api/issues/${issue.id}` — 路径带 id，告诉后端"改第几条"。反引号 + ${} 是字符串拼接
  • headers — 告诉后端"发的是 JSON 格式"
  • body: JSON.stringify({ status: newStatus }) — 要改成什么状态

后端改完之后，前端重新 fetch("/api/issues") 拿一遍最新列表，然后 setIssues(data) 更新页面——跟页面加载时拿数据是一模一样的模式。

关键测试——刷新页面：状态还在！因为改的是后端数据，刷新后前端重新从后端拿，拿到的是改过的版本。对比课开头：同样是刷新，开头数据丢了，现在数据还在。

4.7 新增 Issue 的 API

拿数据和改状态都有了，还缺添加新 Issue。完整代码如下：

app.post("/api/issues", (req, res) => {
  const { label, title, assignee } = req.body;        // 前端发来的数据

  const newIssue = {
    id: issues.length + 1,                              // 数组长度 + 1
    label,
    title,
    assignee,
    status: "待处理",                                    // 默认待处理
  };

  issues.push(newIssue);                                // 加到数组末尾
  res.json(newIssue);                                   // 返回 JSON
});

跟 GET 和 PUT 的结构一模一样：app.方法(路径, 处理函数)。区别是 POST 要从 req.body 里取前端发来的数据，组装好之后 push 进数组，再返回。

4.8 总结

核心概念 一句话
前端 管显示和交互（页面长什么样、用户怎么操作）
后端 管数据和逻辑（数据存在哪、怎么改）
API 前后端之间的约定（路径 + 方法 + JSON）
fetch 前端调 API 的方式

速查卡

后端 & API 速查卡
═══════════════════════════

常见 API 方法：
  GET    → 查数据    → app.get("/路径", (req, res) => { ... })
  POST   → 新建数据  → app.post("/路径", (req, res) => { ... })
  PUT    → 改数据    → app.put("/路径", (req, res) => { ... })
  DELETE → 删数据    → app.delete("/路径", (req, res) => { ... })

前端调 API：
  fetch("/api/xxx")                        → GET，拿数据
  fetch("/api/xxx", {                      → POST/PUT/DELETE，发数据
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ ... })
  })

后端收数据：
  req.params.id  → 路径里的参数（/api/issues/:id）
  req.body       → 前端发过来的 JSON 数据

后端发数据：
  res.json(数据)         → 返回 JSON
  res.status(404).json() → 返回错误码 + JSON
