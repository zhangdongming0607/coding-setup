第 4 课：后端 — 数据存在哪、API 是什么

把 Issue 看板的假数据搬到后端，搞懂前后端分工和 API 通信。

4.1 问题：数据写死了，刷新就没了

上节课我们的 Issue 看板，数据写死在代码里：

const [issues, setIssues] = useState([
  {label: 'Bug', title: '首页加载速度太慢', assignee: '小明', status: '进行中'},
  {label: 'Bug', title: '登录功能异常', assignee: '小明', status: '待处理'},
  // ...
])

点按钮改了状态，刷新页面就没了——因为每次刷新，React 重新执行函数，useState 又从初始值开始。

这就像在白板上写数字，擦掉重画，数字当然没了。

真实的应用里，数据不写在前端代码里。数据存在后端——一个独立的程序，专门管数据。前端要数据就问后端要，前端要改数据就告诉后端去改。

4.2 前端和后端各管什么

用餐厅来类比（第 1 课我们用餐厅讲过请求链路，今天进厨房看看里面怎么运作）：

角色 类比 职责
前端 餐厅大厅 菜单长什么样、桌椅怎么摆、客人怎么点菜——你看到的一切、你的操作
后端 厨房 + 仓库 食材存在哪、今天有没有番茄、菜怎么做——数据存在哪、业务逻辑怎么算
API 传菜窗口 大厅和厨房之间的约定——点菜单怎么写、菜怎么递——前端和后端之间的通信规则 
┌─────────────────┐                        ┌─────────────────┐
│   前端（大厅）    │      API（窗口）        │   后端（厨房）    │
│                 │                        │                 │
│  显示页面        │ ──── 请求 ──────→       │  存数据          │
│  响应用户操作     │                        │  处理业务逻辑     │
│  更新界面        │ ←─── 响应（数据）──      │  返回结果         │
└─────────────────┘                        └─────────────────┘

回忆第 1 课的聊天 App——你点发送，前端通过 API 把消息发给后端，后端返回回复，前端再画到页面上。Network 面板里看到的那些请求，就是 API 调用。

4.3 两个终端：模拟大厅和厨房

前端和后端是两个独立的程序，要分别启动。就像餐厅大厅开门了不代表厨房也在运转——两边都得有人。

在 VS Code 里开两个终端标签页：

终端 角色 命令 端口
终端 1 大厅（前端） cd lessons/lesson-4 && npm run dev localhost:5173
终端 2 厨房（后端） cd lessons/lesson-4/server && npm run dev localhost:3001

两个端口号不一样——5173 是大厅的门牌号，3001 是厨房的门牌号。它们通过 API 互相通信。

4.4 API 长什么样

API 不抽象——在浏览器地址栏输入 http://localhost:3001/api/issues，回车：

[
  { "id": 1, "label": "Bug", "title": "首页加载速度太慢", "assignee": "小明", "status": "进行中" },
  { "id": 2, "label": "Bug", "title": "登录功能异常", "assignee": "小明", "status": "待处理" },
  ...
]

你刚才做的事情，就是手动调了一次 API。跟输入 baidu.com 没有本质区别——都是浏览器去找服务器要东西。区别是 baidu.com 返回网页，这个地址返回纯数据。

API 三要素

要素 点菜类比 例子
路径（URL） 你要的是哪道菜 /api/issues
方法（Method） 你要干嘛 GET（看菜单）、POST（下新单）、PUT（改单）、DELETE（退菜）
数据（JSON） 菜单上写的内容 { "title": "...", "status": "..." }

JSON 就是前后端之间传数据的格式——长得跟第 3 课 useState 里写的数据一模一样。方括号 [] 是数组（一组东西），花括号 {} 是对象（一个东西的详细信息）。

4.5 看代码：后端长什么样

打开 server/index.ts，整个文件做了三件事：开店、定菜单、开门。

第一块：开店

import express from "express";

const app = express();
app.use(express.json());

引入 Express（Node.js 最常用的后端框架，相当于"餐厅管理系统"），创建一个 app。express.json() 告诉它"前端发来 JSON 数据时帮我自动解析"。

这几行是固定写法，不需要记。

第二块：数据

const issues = [
  { id: 1, label: "Bug", title: "首页加载速度太慢", assignee: "小明", status: "进行中" },
  { id: 2, label: "Bug", title: "登录功能异常", assignee: "小明", status: "待处理" },
  // ...
];

眼熟吗？就是第 3 课 App.tsx 里 useState 那段数据，搬到了后端。

多了一个 id 字段——每条 Issue 的身份证号。为什么不用 index？因为你删了第 2 条，后面每条的 index 都变了。id 不会因为删除或排序而改变。

数据存在 JavaScript 变量里，跑在服务器内存中。刷新页面不会丢（刷新只重新加载前端，后端的变量不受影响）。但服务器重启就没了——真实项目会用数据库，今天先用变量把流程跑通。

第三块：API — 获取所有 Issue

// GET /api/issues —— 获取所有 Issue
app.get("/api/issues", (req, res) => {
  res.json(issues);
});

这就是你刚才在浏览器里访问的那个 API。拆开看：

app .get ( "/api/issues" ,  (req, res) => { ... } )
 │    │         │                │
 │    │         │                └── 收到请求后做什么（函数）
 │    │         └── 路径（就是浏览器地址栏输入的那个）
 │    └── 方法：GET = 拿数据
 └── 你的餐厅 app

翻译成人话：当有人用 GET 方法访问 /api/issues 时，把 issues 数组以 JSON 格式返回。

  • req = request，前端发过来的东西
  • res = response，你要回给前端的东西
  • res.json(issues) = 把数组转成 JSON 返回

三行代码，一个完整的 API。

第四块：API — 修改 Issue 状态

// PUT /api/issues/:id —— 修改一个 Issue 的状态
app.put("/api/issues/:id", (req, res) => {
  const id = Number(req.params.id);
  const { status } = req.body;

  const issue = issues.find(item => item.id === id);
  if (!issue) {
    res.status(404).json({ error: "Issue 不存在" });
    return;
  }

  issue.status = status;
  res.json(issue);
});

逐行翻译：

  1. app.put — PUT 方法，表示"修改数据"
  2. "/api/issues/:id" — :id 是占位符。请求 /api/issues/3 时，req.params.id 就是 3。就像跟服务员说"3 号桌"
  3. req.body — 前端发过来的数据（"要改成什么状态"）
  4. issues.find(...) — 在数组里找到对应的那条 Issue
  5. 找不到就返回 404（"找不到"）——第 1 课讲的，4 开头是"你的问题"
  6. 找到了就改 status，把改完的结果返回

餐厅类比：你跟服务员说"3 号桌那道菜改成微辣"——服务员找到单子，改好，回来说"好了"。

第五块：开门营业

app.listen(3001, () => {
  console.log("后端跑在 http://localhost:3001");
});

让服务器开始监听 3001 端口——厨房的门牌号是 3001，前端通过这个门牌号来找后端。

总结：整个后端代码就三件事——开店（创建 app）、定菜单（写 API）、开门（监听端口）。后端比前端简单，它不管页面长什么样，只管"你要什么数据，我给你什么数据"。

4.6 前端对接后端

后端 API 准备好了，现在改前端——让 Issue 看板从后端拿数据。

改动 1：页面加载时从后端获取数据

// 之前：数据写死
const [issues, setIssues] = useState([
  {label: 'Bug', title: '首页加载速度太慢', ...},
  ...
])

// 之后：从后端获取
import { useState, useEffect } from 'react';

const [issues, setIssues] = useState([]);

useEffect(() => {
  fetch("/api/issues")
    .then(res => res.json())
    .then(data => setIssues(data));
}, []);

三行 fetch，三步操作：

代码 做了什么
fetch("/api/issues") 去后端拿数据（就是你在浏览器地址栏输入的那个地址）
.then(res => res.json()) 把返回的 JSON 解析出来
.then(data => setIssues(data)) 放到 state 里 → 页面自动更新

fetch 就是前端调 API 的方式——之前你在浏览器地址栏手动输入 URL，现在代码自动做了。

关于地址：代码里写 fetch("/api/issues")，不是 fetch("http://localhost:3001/api/issues")。因为项目配了一个转发规则（Vite 代理）——前端看到 /api 开头的请求，自动转发到后端 3001 端口。就像大厅有个内部通道直通厨房，不用出门绕后门。

打开 Network 面板验证：能看到 /api/issues 请求，GET 方法，状态 200。

改动 2：修改状态时通知后端

现在能拿数据了，但点"已完成"按钮后刷新，状态还是回去——因为只改了前端 state，没告诉后端。

// 之前：只改前端 state
onChangeStatus={(status) => {
  setIssues(() => {
    const newIssues = [...issues];
    newIssues[index] = { ...issue, status };
    return newIssues;
  })
}}

// 之后：先告诉后端，再重新拿数据
onChangeStatus={(newStatus) => {
  fetch(`/api/issues/${issue.id}`, {
    method: "PUT",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ status: newStatus }),
  })
    .then(() => fetch("/api/issues"))
    .then(res => res.json())
    .then(data => setIssues(data));
}}

跟刚才的 fetch 比，多了几样东西：

  • method: "PUT" — 不再是默认的 GET，而是 PUT（改数据）
  • `/api/issues/${issue.id}` — 路径带 id，告诉后端"改第几条"。反引号 + ${} 是字符串拼接
  • headers — 告诉后端"发的是 JSON 格式"
  • body: JSON.stringify({ status: newStatus }) — 要改成什么状态

后端改完之后，前端重新 fetch("/api/issues") 拿一遍最新列表，然后 setIssues(data) 更新页面——跟页面加载时拿数据是一模一样的模式。

关键测试——刷新页面：状态还在！因为改的是后端数据，刷新后前端重新从后端拿，拿到的是改过的版本。对比课开头：同样是刷新，开头数据丢了，现在数据还在。

4.7 动手：补全"新增 Issue"的 API

拿数据和改状态都有了，还缺添加新 Issue。下面是骨架代码，把 ________ 填上：

// POST /api/issues —— 新增一个 Issue
app.post("/api/issues", (req, res) => {
  // 第 1 步：从前端发来的数据里取出 label、title、assignee
  const { label, title, assignee } = ________;

  // 第 2 步：组装一条新 Issue
  const newIssue = {
    id: issues.________ + 1,
    label,
    title,
    assignee,
    status: "________",
  };

  // 第 3 步：加到数组里
  issues.________(newIssue);

  // 第 4 步：返回给前端
  res.________(newIssue);
});

提示

  • 第 1 步：前端发来的数据存在哪个对象里？（PUT 那个 API 里用过）
  • 第 2 步：数组有几个元素用什么属性？新 Issue 默认什么状态？
  • 第 3 步：往数组末尾加元素的方法？
  • 第 4 步：后端返回 JSON 用什么方法？（GET 那个 API 里用过）

验证

打开浏览器 DevTools → Console 面板，粘贴以下代码：

fetch("/api/issues", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    label: "需求",
    title: "加一个搜索功能",
    assignee: "小红"
  })
}).then(r => r.json()).then(console.log)

成功的话 Console 会打印出新 Issue 的数据。刷新页面，看新卡片有没有出现。

答案

app.post("/api/issues", (req, res) => {
  const { label, title, assignee } = req.body;        // 前端发来的数据

  const newIssue = {
    id: issues.length + 1,                              // 数组长度 + 1
    label,
    title,
    assignee,
    status: "待处理",                                    // 默认待处理
  };

  issues.push(newIssue);                                // 加到数组末尾
  res.json(newIssue);                                   // 返回 JSON
});

4.8 总结

核心概念 一句话
前端 管显示和交互（页面长什么样、用户怎么操作）
后端 管数据和逻辑（数据存在哪、怎么改）
API 前后端之间的约定（路径 + 方法 + JSON）
fetch 前端调 API 的方式

速查卡

后端 & API 速查卡
═══════════════════════════

常见 API 方法：
  GET    → 拿数据    → app.get("/路径", (req, res) => { ... })
  POST   → 加数据    → app.post("/路径", (req, res) => { ... })
  PUT    → 改数据    → app.put("/路径", (req, res) => { ... })
  DELETE → 删数据    → app.delete("/路径", (req, res) => { ... })

前端调 API：
  fetch("/api/xxx")                        → GET，拿数据
  fetch("/api/xxx", {                      → POST/PUT/DELETE，发数据
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ ... })
  })

后端收数据：
  req.params.id  → 路径里的参数（/api/issues/:id）
  req.body       → 前端发过来的 JSON 数据

后端发数据：
  res.json(数据)         → 返回 JSON
  res.status(404).json() → 返回错误码 + JSON
