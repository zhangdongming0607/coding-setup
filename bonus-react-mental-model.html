React 的核心就 3 件事：组件 / Props / State。搞懂这三个，能看懂 AI 写的 90% 的前端代码。

一、组件 = Figma 的 Master Component

你在 Figma 里一定做过这件事——

画一个「消息气泡」的 Master Component，然后在画布上拖出十几个 Instance，每个 Instance 的文字不同，但结构完全一样：圆角、背景色、头像位置，全都继承自那个 Master。

React 的组件就是完全同一个逻辑：定义一次，用多次，每次传不同的数据。

二、Props = Figma 的 Component Properties

你给 Figma 的 Master Component 加了 Properties，比如 variant = filled / outline，label = 任意文字。拖出 Instance 之后，在右侧面板修改这些属性，Instance 就变了——但你没有动 Master 本身。

React 的 Props 就是一模一样的东西。你从外面传数据进来，组件根据这些数据决定怎么显示。

关键规则：Props 只能从外往里传，子组件不能自己改 Props。
就像你只能在右侧面板改 Instance 的属性，不能在 Instance 内部修改 Master Component 的定义。

function MessageBubble({ content, role }) {

  return (

      {content}
    </div>
  );
}

// 使用时传入不同的 Props——就像改 Figma Instance 的属性

三、State = 白板上会变的数据

想象组件有一块白板，白板上写着一些数据。白板上的内容一变，React 就自动重新画这个组件——你不需要手动去改页面上的文字或元素，改白板就够了。这块白板就是 State。

State vs 普通变量：为什么不能用普通变量？

很多人会问：我直接 let count = 0，然后 count++，不行吗？

❌ 普通变量
0

数据改了，React 不知道
页面纹丝不动

✅ State
0

调用 setCount()，React 收到通知
页面立刻更新

State 的本质：你跟 React 之间的通信通道。
你调用 setXxx(新值)，相当于告诉 React：「白板变了，请重新画一下。」普通变量改了，React 根本不知道，页面就不会动。

查看代码长什么样 
// ❌ 普通变量——页面不会更新
let count = 0;
function handleClick() {
  count++;  // 数据改了，但 React 不知道，页面没反应
}

// ✅ State——页面自动更新
const [count, setCount] = useState(0);
function handleClick() {
  setCount(count + 1);  // 通知 React，页面立刻更新
}

四、useEffect = Figma 的「On Load」触发器

你在 Figma 做原型时，可以给一个 Frame 加一个「On Load」交互——页面一打开，自动播放一段动画，或者自动跳到某个状态。不需要用户点任何东西，加载完就自动发生。

useEffect 就是代码里的「On Load」：组件出现在页面上之后，自动执行一段逻辑。最常见的用途是页面一加载就去后端拿数据。

Demo 项目的 App.tsx 里，欢迎消息就是这么来的：

① 组件第一次出现在页面上
→ useEffect 触发
→ 调 fetchWelcome() 发请求给后端
→ 后端返回欢迎文字
→ setMessages() 更新白板 → 页面显示欢迎消息
② 之后再重新渲染，useEffect 不会再次执行（因为 []）

两种常见用法

写法含义Figma 类比
useEffect(() => {}, []) 只在组件第一次出现时执行一次 On Load — 页面打开时触发一次
useEffect(() => {}, [value]) 每次 value 变化时执行 On Variable Change — 某个变量变了就触发

记住一件事就够了：看到 useEffect，就问自己——这段代码是「什么时候自动触发」的？
看 [] 里面有没有东西：没有 = 只触发一次；有变量 = 那个变量变了就触发。

查看代码长什么样 
// 写法一：只在第一次加载时执行（最常见）
useEffect(() => {
  fetchWelcome().then((text) => {
    setMessages([{ role: "assistant", content: text }]);
  });
}, []);  // ← 空数组 = 只执行一次

// 写法二：每次 messages 变化时执行（比如自动滚到底部）
useEffect(() => {
  bottomRef.current?.scrollIntoView({ behavior: "smooth" });
}, [messages]);  // ← messages 变了就执行

五、渲染 = 数据变了，React 帮你重画

每次 State 变化，React 都会走一遍下面这个流程。点下面的 + 按钮，自动走完一遍：

点 + 触发一次 State 更新，看流程如何运转：
0
→ 每次点击，下方 5 步自动逐步高亮 ↓
原理
1
用户操作触发事件

比如点了「发送」按钮，或者点了「+」——某个函数被调用了。

2
调用 setXxx() 修改白板

比如 setMessages([...messages, 新消息])——告诉 React「白板变了」。

3
React 重新执行组件函数

React 收到通知，把整个组件函数重新跑一遍，算出新的页面结构。

4
对比新旧结构，只更新变化的部分

React 不会刷新整个页面，只改那个变了的地方——比如聊天窗口多了一条气泡。

5
页面更新完成

用户看到新的气泡出现了。整个过程通常在 16 毫秒内完成（1 帧）。

第 1 步 / 共 5 步

Figma 类比：就像你改了一个 Master Component 的颜色，所有用到它的 Instance 立刻都更新了——你不需要一个个手动改，Figma 帮你做了。React 也是同一个逻辑，只不过它更新的是浏览器里的真实页面。

八、思考题

不需要打开代码，根据这节课学到的内容，试着回答下面三个问题。答案可以折叠查看。

思考 1：什么时候用 State，什么时候用 Props？

你在设计一个「点赞按钮」组件。按钮当前是否被点赞（true/false）应该放在 State 还是 Props？如果这个按钮是父组件控制的，答案会变吗？

查看参考答案

自己管自己 → State；父组件决定 → Props。

如果点赞状态只属于这个按钮自己（独立的），应该放在 State：const [liked, setLiked] = useState(false)。

如果父组件需要统一管理所有帖子的点赞状态（比如要同步给后端），那就从外面传进来：<LikeButton liked={post.liked} onToggle={handleToggle} />——这时 liked 是 Props，State 在父组件里。

思考 2：页面刷新后数据消失了，为什么？

你用 AI 做了一个 Todo 应用，添加了几条任务，但刷新页面后全没了。你觉得问题出在哪？

查看参考答案

State 只活在内存里，刷新页面就消失了。

React 的 State（白板）是存在浏览器内存中的，页面一刷新，内存清空，State 回到初始值。

要持久化数据，需要：① 存到后端数据库（最常见）；② 或者存到 localStorage（刷新不消失，但换设备就没了）。

所以 AI 给你写的 Todo 应用如果没有后端，任务就只存在当次会话的白板上。

思考 3：为什么要用 useEffect 拿数据，不能直接在组件里写？

有人说：「我直接在组件函数里写 fetch(...) 不行吗？」你怎么回答他？

查看参考答案

可以写，但每次渲染都会重复请求——会出 bug。

组件函数在每次 State 变化时都会重新执行。如果直接在函数里写 fetch，那么每次任何 State 变化（哪怕和这个请求毫无关系）都会重新发一次请求，造成：

  • 无限循环（fetch 完 → setData → 重新渲染 → 再 fetch → ...）
  • 不必要的重复请求，浪费流量

useEffect(() => { fetch... }, []) 只在组件第一次出现时执行一次，解决了这个问题。
