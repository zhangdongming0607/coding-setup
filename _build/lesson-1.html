第 1 课：全局地图 — 代码到底在哪里运行

本课目标：理解"一个网页从输入 URL 到展示内容"的完整过程，分清前端、后端、配置文件各自的职责，并学会用 Chrome DevTools 来验证你的理解。

在浏览器中打开 Demo

课前准备

  • 安装 Chrome 浏览器
  • 安装 VS Code
  • 安装 Node.js（LTS 版本）
  • 安装 Git
  • 把 Demo 项目 clone 到本地并跑起来

1.1 从一个聊天页面说起

打开浏览器，访问 Demo 项目的本地地址 localhost:5173，你会看到一个简单的聊天页面。在输入框里打一句"你好"，点发送，等一会儿就能收到回复。

就这么简单——你发消息，它回消息。但这背后其实有两个程序在配合工作：一个跑在浏览器里负责画页面，一个跑在你电脑上负责处理消息。

这门课不是要把你变成程序员。你已经在用 AI 写代码了，可能也跑通过一些项目。我们要解决的核心问题是：AI 写的那些代码，到底在干嘛？

弄清楚这件事之后，你再看 AI 生成的任何代码，都能知道它属于哪个环节、该怎么检查它对不对。

1.2 浏览器和服务器的对话

你在浏览器里输入一个地址，就像在餐厅下单。整个过程可以分成五步：

  1. 你说了一个名字（URL）——"我要去 xxx.com"
  2. 服务员查座位表（DNS）——把名字翻译成一个具体的地址（IP 地址），你不需要记 IP，记名字就行
  3. 厨房收到订单（服务器）——服务器接到请求，决定返回什么内容
  4. 菜端上来（HTML/CSS/JS）——浏览器收到一堆文件，开始"摆盘"——把页面画出来
  5. 吃的过程中还要加菜（API 请求）——页面加载完之后，JS 代码会再发请求去拿数据，比如聊天消息、用户信息

用一句话概括：浏览器 → 服务器 → 浏览器。所有的网页交互都围绕这个往返展开。

动手验证

  1. 按 Cmd+Option+I（Mac）或 F12（Windows）打开 DevTools，切到 Network 面板
  2. 在聊天框发一条消息，观察 Network 面板中新出现的 chat 请求——这就是前端和后端通信的过程
  3. 刷新页面，观察瀑布流中的几类请求：
    • 第一个请求（document 类型）：浏览器去找服务器要页面——"下单"
    • JS/CSS 文件：页面要长什么样、要有什么功能——"配料"
    • XHR/Fetch 请求（API 调用）：页面画出来之后，代码又去服务器拿数据——"加菜"
  4. 点击一个 API 请求，展开看 Response 里的 JSON 数据——这就是服务器返回的原始数据，前端拿到后把它渲染成你看到的页面

关键心智模型：浏览器 → 服务器 → 浏览器。今天看到的所有东西，都在这个往返里面。

1.3 前端、后端、配置文件

从写代码的视角来看，项目里的文件分三类：

  • 前端代码：跑在浏览器里。负责"页面长什么样"和"用户点了按钮之后做什么"
  • 后端代码：跑在服务器上。负责"数据存在哪"和"业务逻辑怎么算"
  • 配置文件：不跑在任何地方，是告诉工具"怎么打包 / 怎么启动 / 怎么部署"的说明书

Demo 项目文件结构

lesson-demo/
├── client/                         ← 前端代码——跑在浏览器里
│   ├── index.html                  ← HTML 入口——浏览器最先拿到的文件
│   ├── src/
│   │   ├── main.tsx                ← 启动文件——整个前端从这里开始跑
│   │   ├── App.tsx                 ← 根组件——管理消息列表，是所有组件的"大门"
│   │   ├── api.ts                  ← 调后端的函数——前端通过这里跟后端通信
│   │   ├── styles.css              ← 样式——页面长什么样全写在这里
│   │   └── components/
│   │       ├── ChatWindow.tsx      ← 聊天窗口——显示所有消息
│   │       ├── MessageBubble.tsx   ← 消息气泡——一条消息长什么样
│   │       └── ChatInput.tsx       ← 输入框——用户打字和发送的地方
│   ├── package.json                ← 配置——前端用了哪些第三方库
│   ├── tsconfig.json               ← 配置——TypeScript 的设置
│   └── vite.config.ts              ← 配置——打包工具的设置（含代理转发到后端）
│
├── server/                         ← 后端代码——跑在你的电脑上（服务器）
│   ├── index.ts                    ← 服务器入口——接收请求、返回数据
│   ├── replies.ts                  ← 回复数据——模拟 AI 的回复内容
│   ├── package.json                ← 配置——后端用了哪些第三方库
│   └── tsconfig.json               ← 配置——TypeScript 的设置
│

注意：client 和 server 两个文件夹各有自己的 package.json，因为它们是两个独立的程序。前端跑在浏览器里，后端跑在你的电脑上，它们通过 API 请求互相通信。

关键代码解读

我们来看几个核心文件的实际代码，感受一下前端和后端各自在干什么。

前端启动文件 main.tsx——整个前端从这里开始：

import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { App } from "./App";
import "./styles.css";

createRoot(document.getElementById("root")!).render(
  <StrictMode>
    <App />
  </StrictMode>
);

它做的事很简单：找到 HTML 里的 #root 元素，把 App 组件塞进去。从这里开始，整个页面就由 React 接管了。

前端通信文件 api.ts——前端和后端的"桥梁"：

// 发送消息并获取回复
// 用户点"发送"时调用，发一个 POST 请求到 /api/chat
export async function sendMessage(message: string) {
  const response = await fetch("/api/chat", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ message }),
  });

  if (!response.ok) {
    const errorData = await response.json().catch(() => ({}));
    throw new Error(errorData.error ?? `请求失败 (${response.status})`);
  }

  const data = await response.json();
  return data;  // { reply: "...", timestamp: "..." }
}

这就是前端给后端"打电话"的代码。fetch 是浏览器内置的函数，用来发 HTTP 请求。你在 Network 面板里看到的那个 chat 请求，就是这段代码发出去的。

后端入口 server/index.ts——接收请求、返回数据：

// POST /api/chat — 发送消息时调用，返回模拟回复
app.post("/api/chat", (req, res) => {
  const { message } = req.body;

  const reply = getReply(message);
  const now = new Date().toISOString();

  // 故意延迟 500ms，让你在 Network 面板能看到"等待"状态
  setTimeout(() => {
    res.json({ reply, timestamp: now });
  }, 500);
});

后端用 Express 框架搭建。app.post("/api/chat", ...) 的意思是：当有人往 /api/chat 这个地址发 POST 请求时，执行后面的函数。它不关心页面长什么样——那是前端的事。

文件 类型 作用
client/src/App.tsx 前端 描述页面长什么样、消息列表怎么渲染
client/src/api.ts 前端 前端和后端的"桥梁"——用 fetch 给后端发请求
server/index.ts 后端 接收请求、返回数据，不管页面长什么样
vite.config.ts 配置 告诉打包工具怎么运行项目

重要习惯：以后你看 AI 生成的任何文件，第一件事就是判断：这个文件是给浏览器跑的，还是给服务器跑的？这决定了你应该怎么看它、怎么调试它。

1.4 DevTools：你的调试工具

Chrome DevTools 是前端开发最重要的工具，没有之一。你不需要全学，记住三个面板就够应付 80% 的场景。

1. Elements 面板

  • 右键页面上任意元素 → "检查"，可以看到页面的 HTML 结构
  • 鼠标移上去，页面上对应的部分会高亮
  • 可以直接改样式试效果——比如改个颜色，页面立刻变化
  • 注意：这里改了不会保存到代码里，刷新就没了。但它特别适合"先试一下效果"

2. Network 面板

核心就一件事：你的页面发了几个请求、每个请求成功没有、返回了什么数据。

  • 失败的请求会显示为红色
  • 用 "Fetch/XHR" 过滤按钮可以只看 API 请求

HTTP 状态码速记——看第一位数字就够了：

状态码开头 含义 说明
2xx 成功 请求正常完成
4xx 你的问题 请求本身有问题（地址写错了、参数缺了）
5xx 服务器的问题 服务器自己出了错

实用建议：以后页面出问题，第一反应不是看代码，而是打开 Network 看看请求有没有失败。

3. Console 面板

  • 这里会显示代码的报错信息。页面白屏了？来这里看看有没有红色错误
  • 报错信息一般这么读——先看最后一行（哪个文件出了问题），再看红色的那行（错在哪）
  • 代码里写 console.log('hello')，输出就会出现在这里——这是最简单的调试方式

你可以打开 Demo 项目的 Console 面板看看——api.ts 里已经写了一些 console.log，每次发消息都会在这里输出日志，比如 [前端] 发送消息: 你好。

1.5 AI 生成代码的真相

你每天在用 AI 写代码，但你有没有想过，AI 到底是怎么"写"出来的？

它其实是在"联想"

打开任意 AI 工具，输入：

用 JavaScript 写一个函数，判断一个字符串是不是回文

注意观察——回答是一个字一个字蹦出来的，不是一整段一起出现的。这不是动画效果，它真的是在一个一个生成。

AI 做的事本质上跟输入法联想一样：你打了"今天天气"，输入法提示"很好"。AI 就是一个超级强的联想机器——看到你的问题，然后一个词一个词地预测"下一个最可能的词是什么"，直到生成完整的回答。

每次回答都不一样

把同样的问题再问一遍，你会发现写法变了——变量名不一样，实现方式可能也不一样。因为每次预测都有随机性，就像骰子，每次扔结果都不同。所以"昨天让 AI 写好了，今天再问怎么变了"——这是完全正常的。

它会一本正经地编造东西

试试这个提示词：

用 Express 写一个接口，用 req.getBody() 获取请求体

真相是：Express 里根本没有 req.getBody() 这个方法（正确的是 req.body，我们在 Demo 的后端代码里也看到了）。但 AI 很可能会照写不存在的方法，而且看起来一点毛病都没有。这叫幻觉（hallucination）。

核心结论：AI 是你的助手，但 DevTools 是帮你验证它有没有说谎的工具。它说"这个接口会返回数据"，你打开 Network 看一眼就知道了。

1.6 试一试

打开你自己之前做过的 vibe coding 项目（或者我们的 Demo 项目），完成下面的练习。

练习清单

  • 练习 1：打开 Chrome DevTools → Network 面板 → 刷新页面。数一下你的页面一共发了几个网络请求？截图记录。
  • 练习 2：找到一个 XHR 或 Fetch 类型的请求（就是 API 调用），点开它，看 Response 标签页。这个接口返回了什么数据？用一句话描述。
  • 练习 3：打开 VS Code，看你项目的文件树。找出 3 个前端文件和 1 个配置文件，说说它们分别是干嘛的。（如果有后端文件也找出来）
  • 练习 4（加分）：打开 Console 面板，看看有没有报错信息。如果有，把报错信息复制出来，试着读懂它说的是什么。

常见问题：

  • DevTools 打不开：用 Cmd+Option+I（Mac）或 F12（Windows）
  • Network 面板什么都没有：需要先刷新页面才能看到请求
  • 不知道哪个是 API 请求：用面板上的 "Fetch/XHR" 过滤按钮

1.7 本课小结 + 课后作业

本课要点

  1. 请求链路：浏览器 → 服务器 → 浏览器，所有网页交互围绕这个往返展开
  2. 前端 vs 后端：前端跑在浏览器里管页面，后端跑在服务器上管数据，配置文件是工具的说明书
  3. DevTools 三板斧：Elements 看结构、Network 看请求、Console 看报错
  4. AI 的局限：它靠"联想"生成代码，每次结果不同，还可能编造不存在的东西——用 DevTools 来验证

Git 最小流程（交作业用）

Git 是什么、为什么需要它，下节课会详细讲。现在你只需要会三个命令，能把作业提交上去就行：

# 1. 进入课程仓库目录
cd coding-setup

# 2. 告诉 Git "我要存哪些文件"
git add .

# 3. 存一个版本，附一句说明
git commit -m "第一次作业"

# 4. 推到 GitHub 上
git push

简单说：add 是"把文件放到待归档区"，commit 是"正式归档并贴标签"，push 是"传到 GitHub 上"。

课后作业：画一张"项目地图"

  1. 打开你自己做的 vibe coding 项目（或 Demo 项目）
  2. 列出所有重要的文件/文件夹
  3. 每个文件用一句话说明它的作用
  4. 标注哪些是"前端文件"、哪些是"后端文件"、哪些是"配置文件"

格式不限——文本列表、手绘拍照、任何你顺手的工具都行。

提交方式：在课程仓库里创建一个以你名字命名的文件夹，把作业文件放进去，用 git add / commit / push 提交。
