第 2 课：读代码 — 看懂 AI 写了什么

掌握"读代码"的基本方法——找到入口、顺着数据流走，从入口文件一路追踪到页面渲染。

在浏览器中打开 Demo 项目

0:00-0:05 开场 & 回顾

上节课我们建立了一个全局地图——浏览器发请求、服务器返数据、浏览器画页面。你们也学会了用 DevTools 看网络请求和页面结构。

今天我们要打开代码文件，真正去读它。为什么？因为你们用 AI 生成代码之后，遇到最多的情况是：跑起来了但不知道它在干嘛，出了问题不知道从哪里找。

读代码的目标不是逐行理解每个字母，而是能定位——知道问题出在哪个文件、哪个环节。今天教的方法就一个：找入口，顺着数据流走。

0:05-0:15 方法论：找入口，顺着数据流走

读代码跟读一本书不一样。一本书从第一页开始翻就行了。但代码不是线性的——几十个文件摆在你面前，从哪个开始？

方法很简单，两步：

第一步：找入口。 每个程序都有一个"第一行被执行的代码"，这就是入口。找到它，你就有了起点。

第二步：顺着数据流走。 从入口开始，追踪数据是怎么流动的——用户操作产生了什么数据？数据传给了谁？最终怎么变成了页面上的内容？

这两步适用于任何项目，不管是 React、Vue、还是后端代码。

数据流的基本模型：

用户操作 → 事件处理函数 → 修改数据（状态）→ 页面更新
            ↘ 发请求给后端 → 后端返回数据 ↗

你读代码的时候，就是在这条线上找你关心的那个环节。

0:15-0:35 实战：跟我读一遍 Demo 项目

0:15-0:20 第一步：找入口

打开 VS Code，看 Demo 项目的前端代码（client/src/ 文件夹）。文件不多，但从哪个开始看？

记住规律：前端项目的入口，通常叫 main、index、或 app。我们找一找——main.tsx，就是它。

打开 client/src/main.tsx：

import { createRoot } from "react-dom/client";
import { App } from "./App";

createRoot(document.getElementById("root")!).render(
  <App />
);

这个文件就三件事：

  1. 在 HTML 里找到一个叫 root 的容器
  2. 把 App 这个组件塞进去
  3. 页面就出来了

现在我们知道了——所有的内容都从 App 开始。所以下一步，去看 App。

关键概念

  • import：这行代码的意思是"从别的文件里拿东西来用"。就像你在文档里说"参见附件 A"——代码用 import 来引用其他文件里的功能。
  • 组件（Component）：<App /> 看起来像 HTML 标签，但它其实是我们自己写的一个组件。你可以把组件理解为一个可复用的页面零件——按钮是一个零件，输入框是一个零件，整个聊天窗口也是一个零件。

0:20-0:28 第二步：读根组件

打开 client/src/App.tsx，分块来看：

（1）状态声明部分

const [messages, setMessages] = useState<Message[]>([]);
const [loading, setLoading] = useState(false);

这两行是这个组件的状态（state）。什么意思？

你可以把状态想象成一个白板上的数据。页面上要显示什么、现在是什么情况，都写在这个白板上。

  • messages：消息列表，一开始是空的 []
  • loading：正在等回复吗？一开始是 false（没有在等）

关键特性：状态一变，页面自动刷新。你不需要手动去改页面，只要改数据，React 会帮你更新页面。这个下节课会详细讲。

（2）useEffect 部分

useEffect(() => {
  fetchWelcome().then((welcomeText) => {
    setMessages([{ role: "assistant", content: welcomeText, ... }]);
  });
}, []);

这段代码的意思是：页面第一次打开的时候，去后端拿欢迎消息，然后放到消息列表里。

fetchWelcome() 就是上节课我们在 Network 面板里看到的那个 /api/welcome 请求。代码在这里，请求在 Network 面板里——能对上了吧？

（3）handleSend 部分

const handleSend = async (text: string) => {
  // 1. 先把用户消息加到列表里
  setMessages((prev) => [...prev, userMsg]);
  setLoading(true);

  // 2. 发请求给后端，等回复
  const { reply, timestamp } = await sendMessage(text);

  // 3. 把回复加到列表里
  setMessages((prev) => [...prev, assistantMsg]);
  setLoading(false);
};

这就是你点"发送"之后发生的事：

  1. 把你输入的文字加到消息列表里（页面上立刻多一条消息气泡）
  2. 发请求给后端（Network 面板里会出现一个 chat 请求）
  3. 收到回复后，把回复也加到列表里（页面上又多一条气泡）

看到没有——整个流程就是在改 messages 这个状态。状态变了，页面就自动更新了。

（4）render 部分（JSX）

return (
  <div className="app">
    <header className="app-header">
      <h1>Demo 聊天助手</h1>
    </header>
    <ChatWindow messages={messages} loading={loading} />
    <ChatInput onSend={handleSend} disabled={loading} />
  </div>
);

最后这部分描述页面长什么样。看起来像 HTML 对吧？这叫 JSX——React 用它来描述页面结构。

注意两个自定义组件：

  • ChatWindow：显示消息列表。它接收 messages 数据——也就是说，App 把自己的"白板"分享给了 ChatWindow
  • ChatInput：输入框。onSend={handleSend} 的意思是"用户点发送的时候，调用 handleSend 这个函数"

这就是 React 的组件结构：大组件把数据传给小组件，小组件把用户操作传回给大组件。

组件树

App（管家：管数据、管请求）
├── ChatWindow（展示：接收 messages，画出所有气泡）
│   └── MessageBubble（展示：画一条消息）
└── ChatInput（交互：用户输入，触发 onSend）

0:28-0:35 第三步：追一条完整的数据流

我们来追踪一个完整的流程——用户发一条"你好"，到页面显示回复，中间经过了哪些文件。

1. 用户在输入框打字、点发送
   → ChatInput.tsx：handleSubmit() 被调用
   → onSend(trimmed) 把文字传给 App

2. App 收到文字
   → App.tsx：handleSend(text) 被调用
   → 先 setMessages 把用户消息加到列表
   → 调 sendMessage(text) 发请求

3. 发请求给后端
   → api.ts：sendMessage() 用 fetch 发 POST 请求到 /api/chat

4. 后端收到请求
   → server/index.ts：app.post("/api/chat", ...) 处理请求
   → 调 getReply(message) 生成回复
   → 返回 JSON 给前端

5. 前端收到回复
   → api.ts：sendMessage() 返回 { reply, timestamp }
   → App.tsx：handleSend() 把回复加到 messages
   → 页面自动更新，ChatWindow 多了一条气泡

6 个文件串起来了：ChatInput → App → api.ts → server/index.ts → api.ts → App → ChatWindow。

以后遇到 bug，你就沿着这条线找——请求发出去了吗？看 api.ts 和 Network。数据返回了吗？看 server。页面没更新？看 App 的状态有没有变。

0:35-0:40 JS 核心概念速查卡

刚才读代码时你们可能看到很多不认识的写法，这很正常。最常见的整理成一张速查卡，以后遇到了翻出来看一眼就行。

写法 含义
import ... from "..." 从别的文件引入功能，像"参见附件"
export function Xxx() 把这个函数公开，让别的文件能 import
const x = 值 声明一个变量（一个有名字的盒子）
函数名(参数) 调用函数（让它执行）
async / await 等待一个需要时间的操作（比如网络请求）
.then(callback) 另一种等待方式，"等结果出来后执行 callback"

React 特有

写法 含义
<组件名 属性={值} /> 使用一个组件，给它传数据（叫 props）
useState(初始值) 声明一个状态，返回 [当前值, 修改函数]
useEffect(() => {}, []) 页面加载后执行一次的代码
{条件 && <元素>} 条件为真时才显示这个元素
{数组.map(item => ...)} 把数组里的每一项变成一个页面元素

不需要背。读代码的时候认不出来就来查这张卡。用多了自然就记住了。

常见卡点提示

  • 不知道从哪个文件开始？ 找跟关键词相关的文件名，或者用 VS Code 的全局搜索（Cmd+Shift+F）搜关键词。
  • 看不懂 import 路径？ ./ 是当前目录，../ 是上一级目录。
  • 追到 api.ts 就断了？ api.ts 发了一个请求到 /api/chat，那后端谁在处理这个路径？去 server/ 里找。
  • 不理解 req.body？ 前端用 fetch 发送的 JSON 数据，后端通过 req.body 接收。

0:50-0:55 总结

今天的核心就一个方法：找入口，顺着数据流走。

再重复一下步骤：

  1. 找到入口文件（通常叫 main / index / app）
  2. 看入口引用了谁（看 import）
  3. 带着具体的问题，沿着数据流追踪：事件 → 函数 → 请求 → 响应 → 状态更新 → 页面刷新

你不需要一行一行全部读完。读代码是为了回答一个具体的问题——"这个按钮点了之后调了哪个接口？""这段文字是从哪里来的？"有了问题，你才知道该追哪条线。
